No Language Left Behind: Scaling Human-Centered Machine Translation

Angela Fan's lecture explains how Meta's NLLB project pushed machine translation toward 200 languages by treating low-resource translation as a data, evaluation, modeling, and safety problem — not merely a bigger-model problem.

🎓 Angela Fan, Meta AI Research ⏱ ~50 min ▶ Watch
Why this matters

There are roughly 7,000 languages in the world, and thousands have little or no digital text. Translation technology has historically served languages with abundant web data, government proceedings, and commercial translation markets. NLLB asks a more human question: what would it take for people speaking low-resource languages to access information, services, and each other with translation quality they can trust?

Running analogy — a global postal service

Think of multilingual translation as a postal network. High-resource languages are cities with paved roads, labeled addresses, and lots of mail. Low-resource languages are villages with few roads and many local naming conventions. NLLB had to build the map, train address readers, discover which letters correspond, add specialist sorting counters, and audit whether deliveries were safe.

Prerequisites — 30-second refresher

Skim these before the pipeline; each appears immediately.

  • Encoder-decoder / seq2seq MT — an encoder reads the source sentence and a decoder generates the target sentence token by token; translation models are conditional language models.
  • Subword tokenization — words are split into reusable pieces so rare words, scripts, and morphology can still be represented; NLLB must share vocabulary across many languages.
  • Sentence embedding and cosine similarity — a sentence becomes a vector; cosine similarity measures whether two vectors point in the same meaning direction, which powers bitext mining.
  • Mixture of Experts — a sparse model routes each token to a few expert feed-forward networks, adding capacity without activating every parameter; see Note 6 / MoE.
  • BLEU / chrF++ — automatic MT metrics that compare model outputs with references; BLEU uses n-gram overlap, while chrF++ is character-aware and helpful for morphology-rich languages.

1.The missing languages problem

Imagine typing a health notice, school announcement, or immigration form into a translation app and discovering your language is missing — or present but unreliable. Angela Fan frames NLLB from that concrete failure mode: many communities want translation tools, but the data infrastructure behind modern MT was never built evenly.

1.1 Human-centered motivation

The project began with interviews: about 44 native speakers across 36 low-resource languages. The team heard both fear of language decline and excitement about being included in translation systems. That matters because the goal is not a leaderboard row labeled "supports 200 languages." The goal is usable, safe translation for real people.

In the postal analogy, asking speakers is like visiting the villages before drawing highways. You learn where mail actually needs to go, which names and scripts people use, and what counts as a respectful delivery.

1.2 The scaling misconception

Common misconception

"Just train a bigger multilingual model on what's online." That works best where online text already exists. For many low-resource languages, the relevant bitext is scarce, monolingual data may be tiny or script-fragmented, and web text may be noisy, religious-domain-heavy, or absent. Scaling compute cannot translate data that was never collected.

Key equation — the data bottleneck

$$\mathcal{D}_{\text{MT}}=\{(x_i,y_i,\ell_s,\ell_t)\}_{i=1}^{N_{\ell_s,\ell_t}}$$

Here each pair \\(x_i,y_i\\) is a source-target translation, and \\(N_{\ell_s,\ell_t}\\) is the number of pairs for language direction \\(\ell_s\to\ell_t\\). For high-resource directions, \\(N\\) can be huge; for low-resource directions, it may be near zero.

✓ Check yourself — why is "more parameters" not enough for a language with almost no text?

Parameters can memorize and generalize from evidence, but they cannot invent reliable translation conventions, scripts, terminology, or evaluation references without data. NLLB therefore invests in seed data, mining, back-translation, language identification, and human evaluation.

2.Evaluation before training: how do we know it works?

Before building the biggest sorting center, NLLB first built test routes. Evaluation is especially hard for low-resource languages because there may be no established benchmark, no large translator market, and disagreement over standards, dialects, or scripts.

2.1 FLORES-200: a benchmark across 200 languages

FLORES-200 is a human-translated many-to-many MT benchmark. English source documents were translated into 200 languages, including many lower- and mid-resource languages, with document context and broad domains rather than only news. Because every language shares aligned content, researchers can evaluate directions like Chinese→French or Wolof→English, not only English-centric pairs.

2.2 Quality and safety checks

Fan emphasizes that "translation quality" is not one number. FLORES uses automatic metrics such as BLEU and chrF++, but NLLB also uses human evaluation because a five-point German translation rating and a five-point low-resource translation rating are not automatically comparable. Safety adds another axis: some errors are merely awkward, while others change medical advice, distance units, or introduce profanity into benign text.

Key equation — metric as a proxy, not the truth

$$\operatorname{BLEU}(\hat{y},y)\approx\text{n-gram overlap},\qquad \operatorname{chrF++}(\hat{y},y)\approx\text{character F-score}$$

The model output is \\(\hat{y}\\), the human reference is \\(y\\). Both metrics are fast enough for research iteration, but human review is still needed for adequacy, fluency, dialect, script, and safety.

✓ Check yourself — why collect multiple scripts for one language variant?

If a language community writes in more than one script, supporting only one script bakes a social choice into the technology. Multiple-script evaluation helps avoid silently privileging one group of speakers over another.

3.Mining parallel data: finding letters with the same meaning

Parallel data, or bitext, is the fuel of supervised MT: source sentence here, human-equivalent target sentence there. For low-resource languages, there may be little public bitext beyond narrow sources such as religious text. NLLB therefore mines translations from monolingual web corpora.

3.1 Embeddings and nearest neighbors

The mining recipe is concrete. Download and clean web text, identify each sentence's language, split documents into sentences, embed every sentence into a multilingual vector space, then look for nearest neighbors across languages. If an Assamese sentence and an English sentence land near each other in meaning-space, they may be translations.

Key equation — cosine similarity for mining

$$\cos(s,t)=\frac{s\cdot t}{\lVert s\rVert\,\lVert t\rVert}$$

The source sentence embedding is \\(s\\), the target sentence embedding is \\(t\\), \\(s\cdot t\\) is their dot product, and the denominator normalizes away vector length so direction carries the meaning match.

3.2 LASER3 and shared space

Sentence encoders are the address system. If they are bad for Urdu, Telugu, Tagalog, or another low-resource language, mining finds false addresses. NLLB improves LASER-style encoders with masked language modeling and multilingual distillation: a multilingual teacher keeps specialized student encoders in the same vector space while allowing them to focus on language families.

Key equation — distilling a shared embedding space

$$\mathcal{L}_{\text{distill}}=1-\cos\big(e_{\text{teacher}}(x),e_{\text{student}}(x)\big)$$

The teacher embedding \\(e_{\text{teacher}}(x)\\) and student embedding \\(e_{\text{student}}(x)\\) see the same sentence \\(x\\). Minimizing this loss keeps student encoders compatible for cross-language mining.

✓ Check yourself — why not train one unrelated encoder per language family?

Mining depends on comparing vectors across languages. If each encoder learns its own coordinate system, "near" in one space is not comparable with "near" in another. Distillation ties the specialized students back to a common teacher space.

4.The multilingual Transformer: one sorting center, many counters

After data comes the translation model. NLLB uses a multilingual encoder-decoder Transformer: the encoder reads the source, the decoder generates the target autoregressively, and a target-language token tells the model which delivery route to use.

4.1 Encoder-decoder translation

Key equation — conditional translation objective

$$p_\theta(y\mid x,\ell_t)=\prod_{j=1}^{T_t}p_\theta(y_j\mid y_{<j},x,\ell_t),\qquad \mathcal{L}=-\sum_j\log p_\theta(y_j\mid y_{<j},x,\ell_t)$$

The source sentence is \\(x\\), the target language token is \\(\ell_t\\), and \\(y_j\\) is the next target subword. Training minimizes negative log-likelihood of the human or mined target tokens.

Positive transfer is the upside: languages share names, numbers, concepts, syntax fragments, or scripts, so one model can learn reusable translation behavior. Interference is the danger: high-resource languages dominate training, while low-resource languages are overfit or overwritten.

4.2 MoE capacity, curriculum, and back-translation

NLLB scales capacity using a sparsely gated Mixture of Experts. Instead of every token passing through one dense feed-forward block, a router sends a token representation to a small number of expert networks. More counters are available, but each letter visits only a few counters, so compute grows more slowly than total parameters.

Key equation — sparse expert routing

$$h'=\sum_{e\in \operatorname{TopK}(g(h))}p_e(h)\,E_e(h)$$

The hidden token state is \\(h\\), router \\(g\\) selects a few experts \\(E_e\\), and \\(p_e(h)\\) weights their outputs. In NLLB, the challenge is not only adding experts; it is regularizing routing so low-resource languages do not overfit.

The lecture highlights three supporting moves:

  • Gating regularization masks or regularizes the router so MoE capacity does not memorize tiny low-resource datasets too early.
  • Curriculum learning introduces languages at different times: high-resource languages train longer; very-low-resource languages enter later to reduce overfitting.
  • Back-translation creates synthetic source-target pairs by translating monolingual target text backward, adding silver data when human bitext is scarce.
✓ Check yourself — what problem does curriculum solve for Wolof-like low-resource data?

If the model sees the tiny low-resource dataset for the whole training run, it may overfit long before high-resource languages finish learning. Introducing it later gives useful training without repeatedly memorizing the same limited examples.

5.Results, impact, and limits

NLLB reported large gains over prior multilingual MT systems on FLORES-101 and produced the first broad FLORES-200 results. Fan notes that the model improved not only automatic BLEU/chrF++ scores but also human evaluation deltas against a baseline Transformer trained on the same data. The system was open-sourced and helped power real translation experiences.

Impact in one sentence

NLLB is important because it made low-resource MT less like charity coverage at the edge of an English-centric system and more like a first-class multilingual engineering target: data, benchmarks, model capacity, safety, and open tools all moved together.

The limitations are equally important. The lowest-resource languages still suffer from missing monolingual text, dialect and script variation, domain mismatch, and uncertain evaluation reliability. Going from 100 to 200 languages was already hard; going toward thousands demands simpler pipelines, better speech-first support, and deeper community participation.

✓ Check yourself — name one NLLB limitation that more BLEU alone would not reveal.

Examples: a dialect is unsupported, a script variant is missing, a translation is toxic despite word overlap, a benchmark domain differs from everyday use, or the community's preferred language standard was not represented.

6.Worked example — bitext mining by cosine similarity

Suppose we have one source sentence embedding \\(s\\) and three candidate target sentence embeddings \\(t_1,t_2,t_3\\). The mining system scores them by cosine similarity:

Step-by-step with real numbers

$$\cos(s,t)=\frac{s\cdot t}{\lVert s\rVert\,\lVert t\rVert}$$

For candidate \\(t_1\\), suppose \\(s\cdot t_1=8.6\\), \\(\lVert s\rVert=2\\), and \\(\lVert t_1\rVert=5\\). Then

$$\cos(s,t_1)=\frac{8.6}{2\cdot 5}=\frac{8.6}{10}=0.86.$$

The three target candidates have scores \\([0.86,0.42,0.71]\\). A simple mining rule might require both a threshold and a margin:

$$\max_i \cos(s,t_i)\ge 0.80\quad\text{and}\quad \max_i\cos(s,t_i)-\operatorname{2ndmax}_i\cos(s,t_i)\ge 0.10.$$

The best score is \\(0.86\\), the runner-up is \\(0.71\\), and the margin is \\(0.15\\). Since \\(0.86\ge0.80\\) and \\(0.15\ge0.10\\), the pair \\((s,t_1)\\) is selected as mined bitext.

✓ Check yourself — why does a margin criterion reduce false pairs compared with raw cosine?

A raw score can be high for generic sentences or crowded neighborhoods in embedding space. The margin asks whether the best candidate is clearly better than plausible alternatives, so it rejects ambiguous matches like two target sentences that both look similarly close.

✓ Check yourself — if the runner-up score became 0.82, would we mine the pair?

No under this rule. The best score is still 0.86, but the margin is only 0.04, which is below the 0.10 requirement. The system would treat the match as ambiguous.

7.End-to-end trace — one sentence through NLLB-style translation

Now follow one concrete delivery through the sorting center: translating "Wash your hands" into a target language.

StepWhat happensShape/Output
0 · inputSource sentence is prepended or paired with a target-language token that says which language to generate."<2wol> Wash your hands"
1 · tokenizeSubword tokenizer splits the source into reusable pieces, including words, fragments, punctuation, and script-specific units.source ids: T_s tokens
2 · encoderTransformer encoder builds contextual source states; in MoE layers, the router sends token states to selected experts.H_enc ∈ R^{T_s×d}
3 · decoder inputDecoder starts from the target-language token and previously generated target subwords.prefix length 1…T_t
4 · cross-attentionEach decoder step attends to encoder states while self-attending to the target prefix; MoE routing can occur in decoder feed-forward blocks too.p(y_j | y_<j, x, ℓ_t)
5 · generateTarget tokens are produced autoregressively until an end token appears.target ids: T_t tokens
6 · detokenizeSubwords are merged into a readable target sentence.target sentence
What the trace reveals

One model shares knowledge across languages because the encoder-decoder machinery, subword vocabulary, attention patterns, and experts can reuse facts learned from related or higher-resource languages. That is positive transfer. The hard engineering problem is preserving that transfer without letting high-resource traffic drown out low-resource routes.

✓ Check yourself — where does the MoE decision happen in this trace?

Inside selected Transformer feed-forward layers. The router looks at a token hidden state and chooses a small set of experts for that token, in the encoder and/or decoder depending on the model configuration.

Glossary

TermPlain meaning
Low-resource languageA language with limited usable digital data for training or evaluating NLP systems; not necessarily a language with few speakers.
BitextParallel source-target sentence pairs used to train translation systems.
LASER / LASER3Meta sentence-embedding models that map sentences from many languages into a shared vector space for retrieval and mining.
Nearest-neighbor miningFinding candidate translations by searching for target sentence vectors closest to a source sentence vector.
MarginA gap between the best match and runner-up match; used to avoid ambiguous mined pairs.
FLORES-200A human-translated benchmark covering 200 languages for many-to-many MT evaluation.
chrF++A character-level translation metric useful when words have rich morphology or tokenization differs across languages.
Back-translationCreating synthetic parallel data by translating monolingual text backward into the source language.
Sparsely gated MoEA model layer where each token activates only a few expert networks, adding capacity without using every parameter for every token.
InterferenceWhen learning one language or direction hurts another, often because capacity or training data is imbalanced.
