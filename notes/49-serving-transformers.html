Serving Transformers: Lessons from the Trenches

Training gets the glory, but serving is where latency, cost, utilization, and user experience are decided.

🎓 Charles Frye · Modal ⏱ ~1 hr ▶ Watch
Why this matters

The lecture’s thesis is sharp: model weights do not become value until people can use them. That use happens through inference servers, and the server decides the actual experience: how long until the first token appears, how smoothly tokens stream, how many users fit on one GPU, and how many dollars each million tokens costs.

The dangerous misconception is “inference is just a forward pass, easy.” Autoregressive serving is a queueing, memory-management, scheduling, and hardware-utilization problem wrapped around a neural network.

Running analogy

Think of an LLM server as a busy restaurant. Prefill is reading a whole table’s order and prepping ingredients in one burst. Decode is plating one bite at a time for many tables. The kitchen is expensive, so the goal is not “cook one dish beautifully”; it is keep the stoves hot, the pantry organized, and every diner served before they get annoyed.

Prerequisites — 30-second refresher

You will follow the note smoothly if these are fresh. For deeper background, connect this to transformer attention from Notes 1 and the efficient-sequence tradeoffs in Note 44.

  • Autoregressive decoding — the model emits one next token, appends it to the context, then repeats; serving cost scales with both prompt and generated length.
  • Attention & KV-cache — attention needs previous keys and values; caching them avoids recomputing the whole past at every generated token.
  • Compute-bound vs bandwidth-bound — a compute-bound step waits on FLOPs; a bandwidth-bound step waits on moving bytes from GPU memory to compute units.
  • Batching — grouping requests lets the GPU do larger matrix operations, but larger batches can increase per-user latency.
  • Quantization — using fewer bits per value, such as int8, int4, or fp8, reduces memory traffic and capacity pressure, sometimes with quality loss.

1.The serving mindset: workloads before kernels

Charles frames inference engineering from the application backward. A chatbot, a background coding agent, and a bulk document processor all call language models, but they impose different constraints. The chatbot needs smooth interaction; the background agent tolerates seconds or minutes; the data processor wants maximum tokens per dollar.

1.1 Workloads and SLOs

A serving team first defines the workload: queries per second, input-token distribution, output-token distribution, prefix reuse, peak-to-average traffic ratio, and latency budget. Then it reasons per replica: how much can one engine on one machine handle before adding more replicas?

Key equation — user-facing latency

$$T_{\text{last}} \approx T_{\text{queue}} + T_{\text{TTFT}} + N_{\text{out}}\,T_{\text{TPOT}}$$

Here, \(T_{\text{TTFT}}\) is time to first token, \(T_{\text{TPOT}}\) is time per output token, and \(N_{\text{out}}\) is generated length. Tool calls and network hops add more terms, which is why production traces matter.

Common misconception

“Throughput is the only number that matters.” Aggregate tokens/s can look excellent while a user sees stuttering tokens. For interactive systems, tail TTFT and TPOT are part of the product.

✓ Check yourself — which metric matters most for a ChatGPT-like UI?

Mostly per-user output tokens per second after a low TTFT. Aggregate throughput matters for cost, but the felt experience is whether the first token arrives soon and the stream does not stall.

2.The two phases of autoregressive inference

One prompt such as “Explain PagedAttention” creates two very different hardware workloads. First, the server processes the full prompt in parallel. Then it enters a loop: sample one token, append it, sample one token, append it. This asymmetry drives most modern serving systems.

2.1 Why the phases bottleneck differently

Prefill sees many prompt tokens at once, so it can use large matrix-matrix operations and tends to be compute-bound. Decode has only one new query token per sequence per step; it repeatedly streams weights and KV-cache from memory, so it tends to be memory-bandwidth-bound.

Key equation — arithmetic intensity

$$I = \frac{\text{operations}}{\text{bytes moved}}$$

High \(I\) means the GPU can be limited by math throughput; low \(I\) means the GPU waits on memory bandwidth. Decode often has very low \(I\), so expensive tensor cores can sit underused unless batching or speculation increases useful work per byte.

✓ Check yourself — why can prefill be faster per token than decode?

Prefill processes many tokens together, turning work into large dense matrix operations. Decode handles only the next token for each active request, so each step has less compute per byte loaded from GPU memory.

3.The KV-cache: the hidden bill behind long context

Without caching, token 2,000 would recompute attention over tokens 1 through 1,999 every time a new token is generated. So servers store every layer’s previous keys and values. That cache is essential, but it grows with sequence length and batch size.

Key equation — KV-cache size

$$\text{KV bytes} \approx 2 \times L \times H \times d_h \times S \times B \times \text{bytes}$$

The factor 2 is for keys and values. \(L\) is layer count, \(H\) attention heads, \(d_h\) head dimension, \(S\) sequence length, \(B\) active batch size, and \(\text{bytes}\) is storage precision such as 2 bytes for fp16/bf16.

3.1 Paging and batching

Variable prompts and outputs make static allocation wasteful: one short request may reserve a long-request-sized slab. PagedAttention, popularized by vLLM, stores KV blocks in fixed-size pages and maps each sequence to the pages it actually uses. Continuous batching then lets the scheduler add new requests and remove finished requests at each decode step instead of freezing a batch until every sequence ends.

✓ Check yourself — why do long-context workloads hurt batch size?

Each active token position needs KV entries in every layer. Longer contexts consume more KV memory, leaving less HBM for other simultaneous requests even if model weights already fit.

4.Techniques that move the throughput–latency curve

A serving system is usually tuning a curve: more batching improves tokens/s and cost/token, but can hurt per-user latency. The techniques below either reduce bytes moved, increase useful work per step, or make the scheduler waste less GPU time.

4.1 Speculation, quantization, parallelism

  • Speculative decoding — a small draft model proposes several tokens; the large target model verifies them. Accepted runs reduce the number of expensive target-model decode steps.
  • Quantization — int8, int4, fp8, or fp4 weights/KV reduce capacity and bandwidth pressure; application evals must verify quality, especially for long sequences.
  • FlashAttention — optimized attention kernels reduce memory movement during attention, especially important in prefill and long prompts.
  • Tensor parallelism — split large models across GPUs so weights fit and memory bandwidth is pooled; communication overhead becomes the price.
  • Prefix / KV caching — reuse previous prompt computations when many requests share a system prompt, document prefix, or agent trajectory.
  • Observability and profiling — log TTFT, TPOT, queue time, cache hit rate, per-replica utilization, power, temperature, and token IDs. You cannot optimize what you cannot see.
Key equation — speculative speedup intuition

$$\text{effective target steps} \approx \frac{N_{\text{out}}}{1 + \mathbb{E}[A]}$$

If the verifier accepts an average of \(\mathbb{E}[A]\) extra draft tokens per target-model call, the big model runs fewer decode iterations. Real speedup also depends on draft-model cost, batching, and acceptance distribution.

✓ Check yourself — when is speculative decoding most attractive?

When target-model decode is memory-bandwidth-bound and spare compute is available. A cheap, accurate draft model can turn one target-model call into multiple emitted tokens.

5.Worked micro-example — KV-cache size arithmetic

Now put numbers on the hidden bill. Consider a 7B-ish transformer with 32 layers, 32 heads, head dimension 128, and fp16 KV-cache storage at 2 bytes per number.

Step-by-step with real numbers

Per generated or prompt token, per active sequence:

$$ \begin{aligned} \text{KV bytes/token} &\approx 2 \times L \times H \times d_h \times \text{bytes} \\ &= 2 \times 32 \times 32 \times 128 \times 2 \\ &= 524{,}288\ \text{bytes} \\ &\approx 0.50\ \text{MiB/token}\quad(\text{about }0.52\ \text{MB/token}). \end{aligned} $$

Many back-of-the-envelope discussions round this to about 1 MB/token after implementation overheads or wider KV layouts. Using the lecture’s requested rough rule:

$$1.05\ \text{MB/token} \times 2048\ \text{tokens} \times 16\ \text{requests} \approx 34{,}400\ \text{MB} \approx 34\ \text{GB}.$$

That is only KV-cache. A 7B fp16 model’s weights are roughly \(7\times10^9\times2\approx14\) GB, so at long context and batch 16 the cache can dominate what limits batch size.

✓ Check yourself — what does halving precision to fp8 do?

It halves the bytes term from 2 to 1 byte, so the raw KV-cache estimate halves. The 34 GB rough estimate becomes about 17 GB, before allocator overheads and quality/accuracy considerations.

6.End-to-end trace — one request through an LLM server

Follow one user request, “Summarize this 2,048-token document,” through a vLLM-style server. The important state is not only text; it is queue position, KV-cache pages, and active decode batch membership.

StepWhat happensBottleneck
1 · request arrivesHTTP/gRPC request is accepted, tokenized, assigned an SLO budget, and placed into a scheduler queue.Queueing + CPU tokenization
2 · prefill promptThe model processes all prompt tokens in parallel and writes each layer’s keys and values into KV-cache pages.Compute-bound; fills HBM
3 · first tokenLogits are sampled; the first output token is streamed to the client. This determines TTFT.Prefill time + queue delay
4 · decode loopEach step reads weights and past KV, produces one token per active sequence, appends new KV, and joins a continuous batch with other requests.Memory-bandwidth-bound
5 · stream tokensDetokenized text is sent incrementally while the scheduler balances fairness, throughput, and latency tails.TPOT + tail jitter
6 · finish and freeThe request emits end-of-text or hits a limit; its KV-cache pages are returned to the free pool for new requests.Memory reclamation
Why decode is memory-bandwidth-bound

During decode, each active request contributes only one fresh query token, but the GPU must still stream model weights and read relevant KV-cache. That is like the restaurant plating one bite after walking through the whole pantry. Continuous batching adds more plates per trip; paging keeps pantry shelves from being wasted on empty space.

✓ Check yourself — where do PagedAttention and continuous batching act in the trace?

PagedAttention acts when KV pages are allocated, appended, reused, and freed. Continuous batching acts inside the decode loop, where finished requests leave and newly prefilling/ready requests can enter between token steps.

Glossary

TermPlain meaning
PrefillPrompt-processing phase where many input tokens are processed together and KV-cache entries are created.
DecodeAutoregressive generation loop that produces one new token per active sequence per step.
KV-cacheStored keys and values from previous tokens, used so attention does not recompute the whole past.
TTFTTime to first token: delay before the user sees the first streamed output token.
TPOTTime per output token: average or percentile spacing between generated tokens.
PagedAttentionvLLM’s virtual-memory-style method for storing KV-cache in pages to reduce fragmentation.
Continuous batchingScheduler behavior that changes the active batch at each step as requests arrive or finish.
Speculative decodingA fast draft model proposes tokens and a larger model verifies them to reduce large-model steps.
QuantizationRepresenting weights or cache values with fewer bits to reduce memory and bandwidth demand.
Tensor parallelismSplitting a model’s matrix operations and weights across multiple GPUs.
