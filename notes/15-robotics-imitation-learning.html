Transformers for Robotics & Imitation Learning

Ted Xiao's lecture argues that robot learning is shifting from hand-built controllers and online RL toward offline, language-conditioned imitation learning scaled with transformer-style foundation-model ingredients.

🎓 Ted Xiao ⏱ ~50 min ▶ Watch
Why this matters

A kitchen robot does not need a prettier demo video; it needs a policy that turns messy observations into safe actions again and again: camera image + goal → next motor command. Hand-coding every contact-rich case fails, and RL from scratch burns expensive real-world interaction plus reward design. Imitation learning starts from the one resource humans can provide: demonstrations of what good behavior looks like.

Running analogy

Think of the robot as an apprentice chef in a chaotic kitchen. Classic programming gives it a rigid recipe card for one counter. RL says, "try things until dinner is good," which is costly and messy. Imitation learning lets the apprentice watch many chefs; transformer policies try to turn all those watched clips into a reusable kitchen sense.

Prerequisites — 30-second refresher

Skim these before the robot starts moving.

  • Policy \(\pi(a\mid o)\) — a mapping from observation \(o\) to an action distribution over \(a\); here it is the robot's learned controller.
  • RL setup — an agent acts in an environment to maximize reward; the lecture contrasts this with offline imitation from already-collected trajectories.
  • Supervised learning / cross-entropy — train by predicting the expert's label; behavior cloning treats expert actions as labels.
  • Tokenization recap — turn images, language, proprioception, and actions into vectors or discrete IDs so a transformer can process them as a sequence.
  • Decision Transformer one-liner — Note 5 framed RL trajectories as token sequences; robot transformers make a similar move for embodied control.
  • Sim-to-real — train or augment in simulation, then deploy on hardware; useful because real robot data is expensive, but visual/dynamics mismatch can break transfer.

1.Why robots need imitation learning

Start with the kitchen apprentice. If the task is "pick up the Coke can," the robot must notice clutter, choose a grasp, avoid collisions, close the gripper, react if the can slides, and stop when done. The same verb looks different on every table.

1.1 The lecture's historical arc: online RL → offline demonstrations

Ted Xiao describes Google's robotics path as a sequence of hard lessons. Around 2016, the team used a 24/7 KUKA "arm farm" for online RL in bins, including QT-Opt and simulation-to-real work such as CycleGAN-style image transfer. Around 2020, the team moved into office micro-kitchens, tried multitask RL and BC-Z-style imitation, and then consolidated around multi-task imitation learning because it could reach high real-world success and improve with more demonstrations.

The paradigm shift

The key move is to decouple data generation from data consumption. Instead of requiring the robot to learn only from its own current trial-and-error loop, collect a static and growing pile of demonstrations, relabel or merge it, and train larger policies offline.

✓ Check yourself — why is offline data such a big deal for robotics?

Because every real robot action costs wall-clock time, hardware wear, supervision, and safety risk. A static dataset can be trained on many times, mixed with other datasets, relabeled, and scaled with compute without physically re-running every experiment.

2.Behavior cloning: the simplest imitation learner

Before the transformer, imagine a spreadsheet of demonstrations. Each row says: when the camera/proprioception looked like this and the language instruction was this, the expert moved the wrist this way and opened or closed the gripper like that. Behavior cloning simply trains the apprentice to choose the same action.

2.1 The BC objective

Let the dataset contain expert pairs \((o,a)\), where \(o\) is the observation and \(a\) is the demonstrated action. The policy \(\pi_\theta(a\mid o)\) assigns probability to actions. Training minimizes negative log-likelihood of the expert action.

Key equation — behavior cloning

$$\min_\theta \; \mathbb{E}_{(o,a)\sim \text{demo}}\big[-\log \pi_\theta(a\mid o)\big]$$

Here \(\theta\) are the policy parameters, \(\text{demo}\) is the demonstration distribution, and \(-\log \pi_\theta(a\mid o)\) is the cross-entropy loss for making the expert action likely.

In RT-1, the continuous robot actions are discretized into bins, so action prediction can be trained like categorical token prediction. That is why a robotics controller can start to look like a language model: predict the next token, except the token moves a gripper.

2.2 The core problem: covariate shift and compounding errors

Supervised learning assumes train and test examples come from the same distribution. Robotics breaks that assumption. One small action error moves the robot into a state the expert never visited; the policy is now off the demonstration road, so it makes a worse action, and the drift compounds.

Pitfall — tiny errors are not tiny in closed loop

Wrong mental model: "If the policy has only 5% per-step error, a 100-step task should have about five bad actions." In closed-loop control, one early bad action changes all future observations. The robot is no longer sampling expert states, so the loss measured on the demo set can dramatically understate real rollout failure.

Key equation — compounding-error intuition

$$\text{supervised-looking mistakes}\approx \epsilon H,\qquad \text{closed-loop exposure cost}\approx \epsilon\sum_{t=1}^{H}t=\epsilon\frac{H(H+1)}{2}$$

\(\epsilon\) is the per-step chance of making an error on expert-like states, and \(H\) is the horizon. The second expression captures that an error at time \(t\) can contaminate many later steps.

The classic repair is DAgger, dataset aggregation: let the current policy visit its own states, ask the expert what should be done there, add those labeled states to the dataset, retrain, and repeat. In apprentice-chef terms: do not only watch perfect cooking videos; practice, freeze the moments you went off-script, and ask the chef what to do from there.

✓ Check yourself — why does DAgger query the expert on learner states, not only expert states?

Because deployment observations are generated by the learner's own actions. Labeling learner-visited states makes the training distribution closer to the rollout distribution.

3.Transformers as robot policies

Once robot experience is a sequence, a transformer can be the apprentice's memory. RT-1, the lecture's central system, takes recent camera frames and a language instruction, compresses them into tokens, applies a causal transformer, and emits discretized action tokens at about three hertz.

3.1 RT-1 tokenizes vision, language, history, and actions

The lecture gives the concrete recipe: RGB image input is patchified, passed through a FiLM-EfficientNet tokenizer, reduced by TokenLearner from many image patches to a small set of task-relevant tokens, mixed with tokenized language instructions, and fed into a decoder-style transformer. The output is not prose; it is a discretized robot action.

Key equation — closed-loop transformer policy

$$p_\theta(a_t\mid o_{t-k:t},\; g)=\prod_{j=1}^{J}p_\theta(z_{t,j}\mid z_{t,<j},\; \operatorname{Tok}(o_{t-k:t}),\; \operatorname{Tok}(g))$$

\(g\) is the language goal, \(o_{t-k:t}\) is the recent observation history, \(z_{t,j}\) are discrete action tokens such as pose bins or gripper state, and \(J\) is the number of action dimensions/tokens predicted for this control step.

3.2 What scaled in the lecture: tasks, data diversity, interoperability

RT-1 was trained on roughly 130,000 demonstrations over about 700 templated tasks. The striking result in the lecture is not merely that a transformer ran on a robot; it was that RT-1 handled seen/unseen tasks, distractors, new kitchens, and even mixtures of data from different sources, including simulation and an older KUKA arm-farm dataset. Gato and BC-Z appear as comparison baselines, with Gato retrained on the team's robot data rather than using its original dataset.

Scaling lesson from the ablations

The lecture emphasizes task diversity: reducing the number of unique tasks hurt more than reducing the number of demonstrations per task. For the apprentice chef, watching the same onion chop 10,000 more times is less useful than seeing many dishes, tools, counters, lighting conditions, and recovery moves.

✓ Check yourself — why discretize continuous robot actions into bins?

It makes action prediction a categorical token problem, which reuses the stable training machinery of sequence models. The cost is resolution: too few bins produce coarse control; too many bins lengthen or sparsify the prediction problem.

4.Language-conditioned policies and foundation-model scaffolding

The lecture's broader thesis is that robotics should exploit models that are improving anyway. Language becomes a shared API between humans, LLM planners, vision-language perception, and low-level skills.

4.1 SayCan and Inner Monologue: planning above the policy

A language model can propose semantically sensible plans, but it does not know what the robot can actually do from the current kitchen state. SayCan combines two scores: the LLM's likelihood that a skill is useful for the instruction and the robot's affordance/value estimate that the skill is feasible now.

Key equation — SayCan-style skill choice

$$s^*=\arg\max_{s\in\mathcal{S}}\; p_{\text{LLM}}(s\mid G,\text{context})\; V_{\text{robot}}(s\mid o)$$

\(s\) is a candidate skill from the robot's skill library \(\mathcal{S}\), \(G\) is the user's goal, \(p_{\text{LLM}}\) measures semantic relevance, and \(V_{\text{robot}}\) measures whether the robot can execute that skill in observation \(o\).

Inner Monologue adds language feedback loops: scene descriptions, active questions, human or VQA answers, and success/failure detectors can be written back into the planner's context so it can replan after the Coke can falls or the human changes their mind.

4.2 DIAL: scaling instructions without recollecting all data

RT-1's original data used hundreds of templated commands, but natural language is much richer. DIAL-style instruction augmentation labels a small fraction of demos with verbose captions, trains or uses a vision-language model, then pseudo-labels the rest. The same episode "pick Coke can" can become "pick up the red Coke can on the right" or "move the lonely object to the others," adding semantic and spatial supervision without recollecting every trajectory.

Key equation — pseudo-labeled imitation data

$$\mathcal{D}_{\text{aug}}=\mathcal{D}_{\text{template}}\cup\{(o_{1:T},\; \hat g,\; a_{1:T}) : \hat g=f_{\text{VLM}}(o_{1:T})\}$$

\(f_{\text{VLM}}\) is a vision-language labeler, \(\hat g\) is its generated instruction/caption, and \(\mathcal{D}_{\text{aug}}\) is the enlarged language-conditioned imitation dataset.

Systems covered in the lecture

BC-Z supplies the earlier multi-task imitation baseline and templated language conditioning. RT-1 is the main transformer control policy. Gato is discussed as a generalist-agent baseline retrained on the robot data. SayCan and Inner Monologue use LLM/VLM scaffolding for planning and feedback. DIAL uses VLMs to expand language labels for robot demos. The transcript does not substantially cover RT-2 or diffusion policies; those are best read as follow-up systems.

✓ Check yourself — why did the lecture care so much about language labels?

Because language turns task identity into a compositional interface. "Pick Coke can" is close to a one-hot label; "pick the red can on the right" exposes color, object, and spatial relations that can recombine at test time.

5.What still limits transformer imitation policies

The lecture is optimistic, but not a claim that transformers magically solve robotics. The apprentice still needs enough varied, safe practice and a way to recover when the kitchen surprises it.

5.1 Data quality, coverage, and real-world cost

Demonstrations are expensive: RT-1-scale data required many robots, teleoperators, and time. Coverage matters more than repetition. If the dataset never contains a recovery from a slipped can, a policy may not invent that recovery at test time. Noisy labels can be tolerable at scale, as DIAL suggests, but too much false grounding teaches the wrong skill-language alignment.

5.2 Safety and generalization

Robots act in the physical world. A language-conditioned policy must refuse unsafe instructions, avoid collisions, detect failures, and generalize without overclaiming. Multi-embodiment and sim-to-real data help, but different grippers, cameras, action spaces, and dynamics make interoperability a research problem rather than a file-format trick.

Common misconception

"More transformer parameters will automatically make a household robot." The lecture's bottlenecks include context length, real-time inference, data collection throughput, task diversity, affordance coverage, and safety. Larger models help only if the robot can feed them the right tokens and execute their outputs reliably.

✓ Check yourself — name one bottleneck that is not model architecture.

Examples: collecting diverse demonstrations, labeling them with grounded language, real-time inference latency, safe exploration, recovery data, multi-robot action interoperability, or planner affordance coverage.

6.Worked numeric micro-example — compounding error by hand

Now compute the failure mode that makes naïve behavior cloning fragile. Use a tiny per-step error probability \(\epsilon=0.05\) and a horizon of \(H=100\) control steps.

Step-by-step with real numbers

If every step were an independent supervised prediction on expert-like states, the expected number of mistaken actions would be roughly

$$\epsilon H = 0.05\times100=5.$$

But suppose one wrong action pushes the robot off the expert distribution and makes later states unreliable. A simple exposure-cost approximation counts how many future time steps can be affected:

$$\epsilon\sum_{t=1}^{H}t=0.05\times\frac{100\times101}{2}.$$

Compute the triangular number:

$$\frac{100\times101}{2}=5050.$$

Then multiply by the per-step error probability:

$$0.05\times5050=252.5.$$

The units are "contaminated future-step exposures," not literal separate crashes. The contrast is the lesson: supervised counting scales like \(O(\epsilon H)\), while closed-loop compounding can scale like \(O(\epsilon H^2)\).

✓ Check yourself — if you double the horizon to \(H=200\), what happens?

Supervised-looking mistakes double: \(0.05\times200=10\). The compounding exposure becomes \(0.05\times(200\times201/2)=0.05\times20100=1005\). Doubling \(H\) makes the quadratic-style term about four times larger, not merely twice as large.

7.End-to-end trace — one transformer control step

Follow one concrete instruction through an RT-1-style policy. The shapes are illustrative but aligned with the lecture: recent images, language tokens, compact learned visual tokens, transformer context, then discrete action bins.

StepWhat happensState / shape / value
0 · inputRobot sees the table and receives the natural-language goal.6 RGB frames + "pick up the coke can" + proprioception
1 · image patchesEach image is patchified and encoded by a visual tokenizer such as FiLM-EfficientNet.per frame: 81 image patch features
2 · token selectionTokenLearner keeps task-relevant visual tokens so inference stays fast.per frame: 8 visual tokens × d
3 · language and robot stateText tokens and proprioceptive/action-history tokens are embedded into the same model width.goal tokens + joint/gripper state tokens
4 · transformerA decoder-style transformer attends over recent visual history and goal context under a causal mask.context sequence → hidden states (N × d)
5 · action tokensHeads predict categorical bins for end-effector motion, rotation, gripper, and termination.Δpose bins + gripper open/close + terminate?
6 · executeThe selected action is converted to robot control, sent to hardware, and the next camera frame arrives.one 3 Hz closed-loop control step
What the trace reveals

The transformer is not directly "understanding pixels" or "speaking motor torques." The system works because every modality is translated into a shared token sequence, and the final prediction is made discrete enough to train like supervised next-token prediction.

Glossary

TermPlain meaning
Behavior cloningSupervised imitation: train a policy to predict the expert's action from the observed state or sensor input.
Covariate shiftThe train-time observations differ from deployment observations, often because the policy's own mistakes create new states.
DAggerDataset Aggregation: run the learner, ask the expert to label learner-visited states, add them to training, and repeat.
AffordanceAn estimate of what actions or skills are physically possible from the current state.
Action tokenA discrete ID representing one part of a robot command, such as a binned wrist displacement or gripper state.
Language-conditioned policyA controller whose behavior changes according to a natural-language instruction.
TokenLearnerA learned module that selects a small number of informative visual tokens from many image patch features.
Sim-to-realUsing simulated data or training and transferring to physical hardware despite visual and dynamics gaps.
Multi-embodiment dataRobot datasets collected from different bodies, sensors, grippers, or action spaces.
