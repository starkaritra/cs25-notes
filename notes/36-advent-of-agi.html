STANFORD CS25 · TRANSFORMERS UNITED · V5

The Advent of AGI — Agents Toward General Intelligence

Div Garg's thesis: LLMs become socially useful intelligence not by chatting harder, but by closing the loop between perception, planning, action, observation, memory, and correction.

🎓 Div Garg ⏱ ~50 min ▶ Watch
Why this matters

A chatbot can explain how to book a flight. An agent should compare flights, use the browser, fill forms, respect your preferences, stop before payment if needed, recover from errors, and leave an audit trail. Garg's talk asks: if AGI is meant to be useful in everyday life, shouldn't it be able to do work in the same digital world where humans already work?

Running analogy — the digital intern

Think of an agent as a careful digital intern. A chatbot writes advice; an intern looks at the screen, makes a plan, clicks, notices the result, updates the plan, asks for approval at risky steps, and learns your preferences over time. The whole note reuses this analogy: memory is the intern's notebook, tools are their office equipment, and evaluations are supervised trial runs before real responsibility.

Prerequisites — 30-second refresher

You'll follow this note smoothly if these are fresh. Each is one line; skim, and jump to the linked note if a term feels shaky.

  • LLM & prompting — a transformer maps a context of tokens to likely next tokens; agents repeatedly turn those tokens into plans, tool calls, and observations.
  • Agent / tool-use — an agent chooses actions in an environment rather than only producing text; see Note 24 on ReAct-style agents.
  • RL / reward — reinforcement learning improves a policy by scoring trajectories; Garg uses it to describe learning from successful and failed web-task attempts.
  • Long-horizon error — a task with many steps succeeds only if most steps are correct; per-step reliability compounds roughly like \(p^H\).
  • Open-world agents — web agents face changing, underspecified environments; compare Note 20 on open-ended worlds.

1.What AGI looks like, in this talk

Garg opens with a deliberately unsettled question: what is the form factor of AGI? Is it a supercomputer, a 10× better chat interface, a personal companion, or something embedded in daily life? His answer is not a formal definition. It is a product-and-systems framing: AGI-like capability should be able to interface with the digital world more efficiently than humans and accomplish useful goals on our behalf.

1.1 Chatbots → copilots → autonomous agents

Start with the concrete ladder from the lecture. A Level-1/2 system assists while the human remains in control, like autocomplete or a coding copilot. A Level-3 system executes more of the work, while a human monitors and corrects. Level-4 systems operate without an immediate human in the loop but with fallback layers; Garg compared this to Waymo-style remote oversight. Level-5 would be fully autonomous, with no human monitor.

The definition by behavior

In this lecture, "AGI" is less a theorem than a behavioral target: a generally useful system that can understand goals, use tools, operate interfaces, remember context, coordinate with other agents, and improve from feedback. Epistemic status: framing / thesis, not a settled scientific definition.

Common misconception

Wrong model: AGI is "ChatGPT with a bigger brain." Garg's counter-model is "ChatGPT with hands, eyes, memory, permissions, evaluations, and recovery loops." Better reasoning helps, but without action and observation it still cannot finish real digital tasks.

✓ Check yourself — classify the autonomy level

If Cursor suggests a code change while you decide what to apply, that is roughly L2. If an agent writes most of a feature, runs tests, and asks you to approve fixes, that is closer to L3. Fully booking a trip without you watching would require L4/L5 safeguards depending on payment and identity risk.

2.Worked micro-example — action selection by softmax

Before the full architecture, compute one agent decision by hand. Suppose the digital intern sees a web page and proposes three candidate actions. A policy model or critic assigns utilities \([0.7, 0.2, 0.5]\):

  • \(a_1\): click the date picker.
  • \(a_2\): go back to the homepage.
  • \(a_3\): type in the search box.
Step-by-step with real numbers

Use a temperature-1 softmax:

$$P(a_i\mid s)=\frac{e^{u_i}}{\sum_j e^{u_j}}$$

Here \(s\) is the current page state, \(u_i\) is the utility for action \(a_i\), and \(P(a_i\mid s)\) is the probability of choosing that action.

$$e^{0.7}=2.0138,\quad e^{0.2}=1.2214,\quad e^{0.5}=1.6487$$

$$Z=2.0138+1.2214+1.6487=4.8839$$

$$P(a_1)=0.412,\quad P(a_2)=0.250,\quad P(a_3)=0.337$$

The top action is \(a_1\), because it has the highest utility and the highest softmax probability. A stochastic agent might still sample \(a_3\), but a greedy executor would click the date picker.

Why long horizons are unforgiving

If each step succeeds with probability \(p\), and the task needs \(H\) correct steps, a crude independence model gives:

$$P(\text{task success})\approx p^H$$

At \(p=0.9\) for \(H=10\):

$$0.9^{10}=0.3487\approx 35\%$$

Even a "pretty good" action selector can fail most long web tasks. At \(p=0.99\), the same 10-step task is \(0.99^{10}=0.904\), which explains Garg's emphasis on 99%+ reliability before high-stakes deployment.

✓ Check yourself — why is 90% per-step not enough?

Because long tasks multiply failure chances. Ten steps at 90% each gives only about 35% task success; twenty steps gives \(0.9^{20}\approx 12\%\). Agents need recovery, verification, and very high local accuracy.

✓ Check yourself — if \(u_2\) rose to 0.9, what changes?

Then \(a_2\) becomes the top-scored action. The exact probabilities would be proportional to \(e^{0.7}, e^{0.9}, e^{0.5}\), so the homepage action would dominate unless a rule or critic penalized it as a bad plan.

3.Agent architecture: perceive → plan → act → observe

Garg points to the now-standard LLM-agent architecture: the model is the controller, but it needs memory, tools, planning, and actions. A pure text model is like a smart intern trapped in a chat box; an agent loop gives it the screen, tools, and a way to revise after reality answers back.

3.1 Memory, tools, planning, action

  • Memory — short-term context for the current task plus long-term user history and preferences.
  • Tools — calculators, calendars, web search, code execution, APIs, browsers, and computer-control affordances.
  • Planning — decomposition into subgoals, reflection, self-criticism, and recovery when an action fails.
  • Action — clicking, typing, navigating, calling functions, sending messages, or using APIs on the user's behalf.
Architectural invariant

The agent is not one model call. It is a closed-loop system: model call → action → environment feedback → new model call. That feedback loop is what lets the digital intern correct itself rather than merely hallucinate a final answer.

✓ Check yourself — where does self-correction live?

Self-correction lives in the loop between observation and planning. After an action changes the page, the agent observes the new state, compares it with the goal, and revises the next action or backtracks.

4.End-to-end trace — a web agent books a flight

Now follow one concrete task all the way through. The state column tracks what the agent knows or changes at each step.

StepWhat happensOutput
0 · goalUser says: "Book a flight from San Francisco to Seattle next Friday evening; ask before payment."goal + constraint
1 · perceive pageAgent opens a travel site and reads DOM fields plus screenshot: origin, destination, date, passengers, search button.state: form visible
2 · plan next actionPlanner decomposes the task: fill route → choose date → filter times → compare fares → request approval.subgoal: fill route
3 · actAgent types "SFO" in origin, "SEA" in destination, and clicks the date picker.tool calls: type/click
4 · observe resultThe site opens a calendar; the chosen date is not visible, so the agent scrolls or advances one week.new DOM + screenshot
5 · re-planIt updates the plan: choose the date, then filter departure time to evening.subgoal: select date

What the trace reveals

Closing the perceive-act loop is what makes this an agent. A one-shot LLM can describe the booking process; a web agent repeatedly tests its plan against the live interface and updates its next move.

✓ Check yourself — where should the agent pause?

At irreversible or high-risk steps: payment, cancellation, sending messages, revealing private information, or anything that affects another person. That pause is a permission boundary, not a failure.

5.Training agents: learn from traces, not only answers

Garg's strongest technical point is that general LLMs are often not trained on interface-control distributions. That explains why impressive chat models can still fail simple web tasks. To improve, agents need interactive data: trajectories showing what went right, what went wrong, how to backtrack, and which action should have been chosen.

5.1 Agent Q: search + critique + DPO

The lecture's concrete example is Agent Q, a self-improving agent system that combines Monte Carlo Tree Search, self-critique, and preference optimization. In the OpenTable-style reservation example, the agent may go to the wrong page, recognize the mistake, backtrack, choose the correct restaurant/date/time, and turn those successes and failures into training data.

Planning objective — expected future reward

In search-based RL language, a state value estimates future payoff:

$$V(s)=\mathbb{E}\left[\sum_{t=0}^{H} \gamma^t r_t \mid s_0=s\right]$$

Here \(s\) is the current page state, \(r_t\) is the reward after step \(t\), \(\gamma\in[0,1]\) discounts distant rewards, and \(H\) is the horizon. MCTS uses sampled futures to decide which branch is worth exploring.

Preference update — the intuition behind DPO

DPO trains a policy to prefer a better trajectory \(y_w\) over a worse trajectory \(y_l\) for the same state or prompt \(x\):

$$\mathcal{L}_{\text{DPO}}=-\log \sigma\left(\beta\left[\log\frac{\pi_\theta(y_w\mid x)}{\pi_{ref}(y_w\mid x)}-\log\frac{\pi_\theta(y_l\mid x)}{\pi_{ref}(y_l\mid x)}\right]\right)$$

Read it as: make the updated policy \(\pi_\theta\) assign relatively more probability to the winning trace than the losing trace, compared with a reference policy \(\pi_{ref}\). \(\beta\) controls how strongly to enforce the preference.

Transcript-specific anchor: Garg reports that, for an OpenTable reservation task, a base setup around roughly 18.6–20% improved dramatically under Agent Q-style learning, reaching about 95.4% with the combined MCTS, DPO, and self-critique recipe. He also notes the run used 50 H100s for less than a day. Treat these as reported experimental results from the talk/paper, not universal guarantees for every website.

✓ Check yourself — why does a critic help?

The actor proposes possible actions; the critic gives feedback before or after acting. That feedback can rank "choose the correct date" above "go to the homepage," producing cleaner training signals than only seeing final success or failure.

6.Evaluating and trusting agents

The talk repeatedly returns to trust. Garg's team built RealEvals, described as a sandbox with cloned versions of popular websites such as Airbnb, Amazon-like shopping, DoorDash-like delivery, LinkedIn-like workflows, email, and calendar environments. The goal is to test agents on realistic web tasks before letting them touch real accounts.

6.1 API agents vs human-like computer-use agents

ApproachWhy it is attractiveWhere it breaks
API agentsMore controllable, easier to constrain, safer for well-specified services.Public APIs cover only part of the internet; every API has its own schema, permissions, and version changes.
Human-like UI agentsCan operate existing software by clicking and typing, including login/payment flows humans already use.Harder to guarantee behavior; visual/DOM changes, loops, and hidden state can cause failures.
Hybrid agentsUse APIs where reliable and UI control where no API exists.Need routing, permissioning, monitoring, and fallback when one channel fails.

Garg reports early benchmark numbers that make the reliability gap concrete: GPT-4o was around 14% success in one RealEvals setup; OpenAI's computer-use model behind Operator was not yet strong across many tasks; and the best closed models in the benchmark were still far from production-grade reliability. The exact leaderboard can change, but the lesson is stable: impressive chat capability does not automatically imply robust long-horizon action.

Common misconception

Wrong model: "If a model can solve hard math, it can click through any website." Web control needs grounding, state tracking, error recovery, permission boundaries, and tolerance for messy interfaces. Reasoning is necessary but not sufficient.

Deployment checklist
  • Offline benchmark on realistic cloned tasks.
  • Regression tests for prompt/model changes.
  • Audit trail of every action and observation.
  • Human override for high-risk or ambiguous steps.
  • Budget controls to prevent loops from burning compute or money.
✓ Check yourself — why can one bad real-world mistake kill trust?

Because users experience agent failures as delegated harm: the wrong email sent, a bad transaction, a deleted file, or a missed booking. Unlike a sandbox, the real world does not offer unlimited harmless retries.

7.Memory, personalization, and agent communication

Garg's processor analogy extends naturally: if the transformer is a CPU over language tokens, then the agent also needs RAM-like working context and disk-like long-term memory. Long-term memory enables personalization: allergies when ordering food, seat preferences when flying, favorite brands when shopping, and implicit preferences learned from repeated behavior.

7.1 MCP, A2A, and lossy handoffs

Single agents are limited. Multi-agent systems promise parallelization and specialization: a spreadsheet agent, a browser agent, a Slack agent, a coding agent. But Garg emphasizes the failure mode: natural-language communication is lossy. With \(n\) agents, naive pairwise coordination can create about \(n^2\) communication paths, and each handoff can drop or distort information.

Communication overhead

A fully connected team of \(n\) agents has approximately

$$\frac{n(n-1)}{2}=O(n^2)$$

possible pairwise channels. More channels can mean more parallelism, but also more chances for mismatch, stale state, duplicated work, and hidden assumptions.

The talk names two protocol directions: MCP, Anthropic's Model Context Protocol, as a standard wrapper for tools and APIs; and A2A, Google's Agent2Agent protocol, for agent interoperability. Epistemic status: emerging infrastructure. These protocols may reduce ad hoc integration pain, but they do not by themselves solve grounding, intent, or safety.

✓ Check yourself — what should long-term memory store?

Store durable, useful preferences with provenance and user control: allergies, travel preferences, recurring workflow choices. Do not silently store sensitive data or one-off instructions as if they were permanent identity facts.

8.The trajectory: useful AGI as reliable delegation

Garg's forecast is that humans increasingly become managers of agent systems: we set goals, constraints, and feedback, while specialized agents execute. Coding agents are the early example; browser, voice, API, and research agents are plausible next domains. Epistemic status: speaker opinion / forecast. The lecture does not prove a timeline; it argues for a direction.

The thesis in one sentence

The path from LLMs to everyday AGI runs through agents that can act in real digital environments, learn from failures, communicate through robust protocols, and earn trust through evaluation and permissioned deployment.

Forecast humility

"Agents will lead to AGI" is not a settled empirical law. It is a research bet. The strongest evidence in the talk is not a philosophical argument; it is the practical observation that action, memory, and feedback reveal bottlenecks chat-only benchmarks hide.

✓ Check yourself — what would falsify the strong version of the thesis?

If agents remain brittle even after task-specific RL, strong evaluation, better foundation models, and protocolized tool use, then "agentic form factor → AGI" would need revision. Another falsifier: useful general intelligence emerges through a non-agent interface that solves delegation without direct action loops.

Glossary

TermPlain meaning
AgentA system that observes an environment, chooses actions, and updates its behavior from the results.
Computer-use agentAn agent that operates a graphical interface by clicking, typing, reading screenshots/DOM, and navigating like a human.
Tool callingLetting a model invoke a calculator, API, browser, database, code runner, or other external capability.
Short-term memoryThe working context for the current task: goal, recent observations, intermediate plan, and constraints.
Long-term memoryPersistent user or world knowledge retrieved across sessions, ideally with consent, provenance, and editing controls.
MCTSMonte Carlo Tree Search: a planning method that samples possible future action paths and uses their outcomes to guide choices.
DPODirect Preference Optimization: a way to train a model from preferred vs dispreferred outputs or trajectories.
MCPModel Context Protocol: a standard interface for exposing tools and data sources to agentic models.
A2AAgent2Agent protocol: Google's proposed protocol for interoperable communication between agents.
Permission gateA deliberate pause before high-risk actions such as payment, deletion, external messages, or sensitive data access.
