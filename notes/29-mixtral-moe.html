Demystifying Mixtral of Experts

Albert Jiang explains how Mixtral 8×7B turns a transformer’s feed-forward block into a sparse expert marketplace: roughly 47B parameters live in memory, but each token activates only about 13B of them.

🎓 Albert Jiang / Mistral AI & University of Cambridge ⏱ ~50 min ▶ Watch
Why this matters

Dense scaling makes a costly bargain: to give the model more knowledge, every token must pass through more parameters. Mixtral 8×7B changes the bargain. It is an open sparse Mixture-of-Experts model that can match or beat much larger dense baselines while doing roughly dense-13B-sized active work per token. If the general MoE mechanism feels new, revisit Note 6: MoE & Switch; this note focuses on what Mixtral adds in architecture, interpretation, and serving.

Running analogy

Think of each transformer layer as a hospital floor. A dense model sends every patient through the same general clinic. Mixtral keeps eight specialist clinics on the floor, lets triage choose the top two clinics for each patient, and blends their opinions. The hospital must still rent space for all clinics, but any one patient visits only two.

Prerequisites — 30-second refresher

You will follow the lecture smoothly if these five pieces are fresh.

  • Mixture-of-Experts / routing — a router sends each token to a small subset of expert subnetworks; see Note 6: MoE & Switch for the generic mechanism.
  • Softmax & top-k — softmax turns scores into weights; top-k keeps only the largest few candidates.
  • Transformer FFN block — after attention, each token independently passes through an MLP/FFN; Mixtral replaces this block with expert FFNs.
  • Parameter vs compute distinction — parameters describe how much model capacity must be stored; active compute describes how much work a token actually uses.
  • Decoder-only transformer — Mixtral keeps the autoregressive, attention-then-FFN layer shape of Mistral 7B and changes the FFN part.

1.The sparse-compute bargain

Start with one token in the sentence “The proof follows by induction.” In a dense transformer, that token visits the same FFN as every other token. In Mixtral, the token first passes through attention, then a router chooses two expert FFNs out of eight for that token in that layer. Only after seeing the concrete routing trick do we name the architecture: a sparse Mixture-of-Experts decoder-only transformer.

1.1 Dense layer, sparse FFN

Albert begins by reviewing the Mistral 7B layer: RMSNorm, attention with grouped-query attention, residual connection, RMSNorm, FFN, residual connection. Mixtral keeps this skeleton. The surgical swap is:

  • Attention stays shared: every token uses the same attention module in the layer.
  • FFN becomes eight expert FFNs: each expert is a SwiGLU-style feed-forward network.
  • Router is per token, per layer: the model chooses the top two experts for that token at that layer.
Key equation — from dense FFN to sparse MoE FFN

$$\text{dense: } y=\operatorname{FFN}(x) \qquad\quad \text{Mixtral: } y=\sum_{i\in\operatorname{top\text{-}2}(x)} g_i(x)\,E_i(x)$$

Here \(x\in\mathbb{R}^{d}\) is one token’s hidden vector, \(E_i\) is expert \(i\)’s FFN, and \(g_i(x)\) is the router’s normalized gate weight for the selected expert.

1.2 What “8×7B” really means

The model name is easy to misread. It does not mean there are eight complete 7B models and the token runs through two whole models. Each of Mixtral’s 32 transformer layers contains eight FFN experts, while attention, embeddings, routers, and other parts are shared.

Crisp distinction — capacity is not active compute

Mixtral 8×7B has roughly 46.7B ≈ 47B total parameters that must be stored. But a token activates only the shared path plus 2 of 8 FFN experts per layer, totaling about 12.9B ≈ 13B active parameters. The mental model is: 47B-model memory/capacity, dense-13B-ish per-token compute.

Common misconception

“8×7B means 56B active parameters.” No. Attention and router weights are shared, and only two experts fire for a token in each MoE layer. Also, “13B active” is not the whole serving cost: dynamic routing adds communication and batching complications.

✓ Check yourself — if one layer has 8 experts and top-2 routing, what fraction of that layer’s experts run for one token?

Two out of eight, so \(2/8=25\%\) of the expert FFNs in that layer run for that token. The other six still occupy memory and may serve other tokens in the batch.

2.Top-2 routing inside a Mixtral layer

Imagine the hospital triage desk sees a patient with symptoms that are mostly neurological but partly metabolic. It does not send the patient to every clinic. It scores each clinic, picks the best two, then blends their diagnoses. The router does the same thing with token vectors.

2.1 Router logits to weighted experts

For one token hidden vector \(x\), the router computes eight scores. Then it keeps the top two, softmaxes over just those selected scores, runs those two expert FFNs, and adds their outputs with the gate weights.

Key equation — Mixtral top-2 gate

$$z = W_r x, \qquad S=\operatorname{top\text{-}2}(z), \qquad g_i=\frac{e^{z_i}}{\sum_{j\in S}e^{z_j}}\;\;\text{for }i\in S$$

\(W_r\) is the router matrix, \(z\in\mathbb{R}^{8}\) are expert logits, \(S\) is the selected pair of expert indices, and \(g_i\) are the two gate weights that sum to 1.

Key equation — selected expert output

$$y=\sum_{i\in S}g_i\,E_i(x)$$

Each \(E_i(x)\) is a full FFN output vector of the same hidden size \(d\), so the weighted sum also has shape \(d\).

2.2 Why the FFN is the natural place

The lecture’s useful heuristic is: attention often behaves like token-to-token communication or algorithmic routing, while FFNs store much of a transformer’s learned factual and procedural knowledge. MoE-fying the FFN therefore increases the model’s stored capacity where knowledge is plausibly concentrated, without forcing every token to use all of it.

Why MoE-fy MLPs first?

Making the FFN eight-way wider is like hiring many specialist clinics. Knowledge-heavy benchmarks such as MMLU benefit strongly in the lecture’s plots. MoE-fying attention is possible research territory, but Albert notes stability and systems questions, especially under bf16 training.

✓ Check yourself — what stays dense/shared in Mixtral even though the FFN is sparse?

The attention block, residual path, normalizations, embeddings, and router machinery are shared. Sparsity is primarily in the feed-forward expert block.

3.Worked example — top-2 routing by hand

Numbers make the router less mysterious. Suppose one token reaches a Mixtral layer and the router emits logits for experts \(E_1\) through \(E_8\):

Step-by-step with real numbers

$$z=[3.1,\;0.2,\;2.8,\;-1.0,\;0.5,\;1.9,\;0.0,\;0.7]$$

The two largest logits are \(3.1\) and \(2.8\), so the selected experts are \(S=\{1,3\}\). Softmax over only these two scores:

$$g_1=\frac{e^{3.1}}{e^{3.1}+e^{2.8}}=\frac{1}{1+e^{-0.3}}\approx\frac{1}{1+0.7408}\approx0.574$$

$$g_3=\frac{e^{2.8}}{e^{3.1}+e^{2.8}}=\frac{0.7408}{1+0.7408}\approx0.426$$

The MoE output is therefore:

$$y\approx0.574\,E_1(x)+0.426\,E_3(x)$$

The other six experts receive gate weight zero for this token in this layer.

✓ Check yourself — for a 5-token sequence, how many expert FFN calls happen in one Mixtral layer?

Top-2 routing means \(5\times2=10\) expert-token calls in that layer. Those 10 calls may touch fewer than 8 distinct expert modules if several tokens choose the same experts; across 32 layers the path has \(5\times2\times32=320\) expert-token dispatches.

✓ Check yourself — if expert 6’s logit rose from 1.9 to 3.4, which experts would win?

Experts \(E_6\) and \(E_1\) would win because \(3.4\) and \(3.1\) are the top two logits. Their weights would be \(g_6=1/(1+e^{-0.3})\approx0.574\) and \(g_1\approx0.426\).

4.End-to-end trace — one token through one Mixtral layer

Now follow a single token hidden state through one layer. The exact hidden dimension is model-config dependent; for Mixtral 8×7B the common configuration uses \(d=4096\), but the shape logic below is the point.

StepWhat happensShape/active-params
0 · token hiddenThe token enters the layer after the previous residual stream.x ∈ R^d
1 · attentionRMSNorm, grouped-query/sliding-window attention, output projection, then residual update.R^d → R^d · shared dense path
2 · router logitsThe post-attention token vector is multiplied by a router matrix.R^d → R^8
3 · top-2 selectKeep the two largest router logits and softmax over just those two.8 scores → 2 gates
4 · expert FFNsRun exactly two selected SwiGLU FFNs for this token.2 of 8 experts · ~13B active over full model
5 · weighted combineAdd the two expert outputs using their gate weights.g_a E_a(x)+g_b E_b(x) ∈ R^d
6 · residual outputAdd the MoE result back into the residual stream and pass to the next layer.R^d → R^d
Total tallyAll experts must be available, but the token only walks one sparse path.~47B stored · ~13B active/token
What the trace reveals

Sparse activation decouples knowledge from per-token cost. The hospital owns all eight clinics on every floor, so its total expertise is large. But the patient visits two clinics per floor, so the per-patient work is much smaller than “run all clinics.”

✓ Check yourself — where does the vector shape return to \(d\)?

After attention the token remains in \(\mathbb{R}^{d}\). Each selected expert also outputs \(\mathbb{R}^{d}\), and their weighted sum stays in \(\mathbb{R}^{d}\), so the residual stream can continue unchanged.

5.What the experts learn

The tempting story is: one expert for code, one for math, one for medicine, one for French. Albert’s interpretation section pushes against that story. The router gives a rare, discrete-ish signal about which expert handled which token, but the signal does not reveal tidy human-readable departments.

5.1 Not clean topics

The analysis in the talk looked at validation splits from the Pile and compared expert selection across layers 0, 15, and 31. The observed pattern was cautious:

  • Layer 0 looked fairly uniform, consistent with shallow syntactic/token-level processing.
  • Layer 15 showed a suggestive spike: one expert was selected more for DeepMind Mathematics and next most for GitHub, hinting at a math/code feature — but Albert explicitly calls this speculative.
  • Layer 31 moved closer to uniform again, with some arithmetic-looking spikes but no clean domain story.
Common misconception

“The experts should specialize in human topics.” If two experts owned all code, then code-heavy batches would overload them while other experts sat idle. The learned features may be syntactic, positional, frequency-based, or linear combinations of concepts that are efficient for the model but alien to our labels.

5.2 Temporal locality

The strongest clean behavior was temporal locality: consecutive tokens often reuse experts more than chance. For first-choice routing, random chance is \(12.5\%\) with eight experts; in middle layers the same-expert rate was close to double random. When allowing first-or-second choice, the rate was also significantly above the random baseline. In hospital terms, adjacent patients from the same ambulance often go to the same clinics.

Interpretation opportunity

Attention weights ask “which earlier tokens did this token look at?” Router choices ask “which parameters did this token use?” That is a different interpretability handle, but the handle is not a full explanation.

✓ Check yourself — why would perfectly topical experts be bad for serving code-heavy traffic?

If only the “code experts” were selected, they would become bottlenecks while other experts did no work. Efficient serving wants useful specialization and balanced load across experts.

6.Systems and results

Mixtral’s sparse path gives impressive performance-per-active-parameter, but serving it is not the same as serving a dense 13B model. The model has to keep the whole hospital available.

6.1 Memory, communication, batching

The hard systems truth is: all experts must be in memory for fast inference, even though a token computes through only two per layer. That makes Mixtral attractive in data centers with high batch sizes and expert parallelism, but less natural on memory-constrained edge devices.

  • Memory-bound serving — loading ~47B parameters is the footprint; quantization can help, but the inactive experts are not free.
  • Expert parallelism — place different experts on different GPUs or nodes so many tokens can be processed concurrently.
  • Communication overhead — routing tokens to experts across devices costs bandwidth and synchronization; with many experts or cross-node routing, this can dominate.
  • Batching tradeoff — high batch sizes let different experts stay busy; tiny batches may not amortize routing and communication.
  • Sliding-window attention — inherited from Mistral 7B, lower layers attend over local windows while deeper layers propagate information, helping long-context efficiency alongside grouped-query attention.
Serving cost mental model

$$\text{latency} \not\propto \text{active parameters alone}$$

Active parameters predict compute roughly, but memory footprint, token dispatch, GPU-to-GPU communication, load imbalance, and batch shape all matter.

✓ Check yourself — why can Mixtral be worse than a dense model on a memory-starved edge device?

Because the edge device must store or page the full expert set. Even if each token uses only ~13B active parameters, fast inference wants the ~47B total parameters available.

6.2 Benchmarks and open weights

The headline result is not “MoE is magic”; it is “capacity and compute can be separated enough to move the Pareto frontier.” The lecture reports Mixtral 8×7B as a strong open model: around 12.9B active parameters per token, 32k context, Apache 2.0 release, strong multilingual performance, and competitive results against Llama-2-70B and GPT-3.5-style baselines at much lower active compute.

Why open weights matter here

The routing analysis, expert-removal “treasure hunt,” quantization attempts, and serving experiments became community research because people could inspect and modify the model. Open sparse MoEs are not only products; they are laboratories.

✓ Check yourself — what is the safest one-sentence claim about Mixtral’s result?

Mixtral 8×7B stores far more total capacity than a 13B dense model while activating roughly 13B parameters per token, producing strong open-model performance with nontrivial memory and routing costs.

Glossary

TermPlain meaning
Mixtral 8×7BMistral AI’s open sparse MoE language model with roughly 47B total parameters and about 13B active parameters per token.
Sparse Mixture-of-ExpertsA model layer with many expert subnetworks where only a small subset is activated for each input token.
Router / gateThe learned module that scores experts for a token and assigns gate weights to the selected experts.
Top-2 routingA routing rule that chooses the two highest-scoring experts for a token.
Expert FFNOne feed-forward network inside the MoE block; in Mixtral there are eight per transformer layer.
Active parametersThe parameters actually used in the computation path for a token.
Total parametersAll parameters that must be stored to represent the model, including inactive experts.
Expert parallelismA serving/training strategy that places different experts on different devices so routed tokens can be processed in parallel.
Temporal localityThe tendency for nearby/consecutive tokens to choose overlapping experts more often than random chance.
Sliding-window attentionAn attention pattern where tokens attend locally in lower layers while deeper layers propagate information over longer spans.
