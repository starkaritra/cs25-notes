The Ultra-Scale Playbook: Training Transformers on Thousands of GPUs

Frontier training is not “buy more GPUs”; it is the art of fitting the model in memory while keeping thousands of accelerators doing useful work.

🎓 Nouamane Tazi · Hugging Face ⏱ ~1 hr ▶ Watch
Why this matters

A modern model can have hundreds of billions to trillions of parameters, train on trillions of tokens, and target million-token contexts. On one GPU, the model states may not fit; even if they did, one training run would take far too long. Ultra-scale training is therefore a systems discipline: fit the model, feed the data, and hide communication behind compute.

Running analogy

Think of a giant restaurant kitchen. Data parallelism gives many cooks the same recipe but different orders. Tensor parallelism splits one huge chopping task across cooks at the same station. Pipeline parallelism puts appetizer, entrée, and dessert on different stations. ZeRO/FSDP assigns pantry shelves so no cook stores the whole inventory. Scaling works only when handoffs are fast enough that cooks are not waiting with empty hands.

Common misconception

“Just add more GPUs and training gets linearly faster.” That picture ignores the handoffs: all-reduce, all-gather, reduce-scatter, send/receive, and all-to-all. At scale, the network often becomes the limiter, so the real goal is high utilization, not merely high GPU count.

Prerequisites — 30-second refresher

Skim these before the cluster diagrams; each term is used immediately below.

  • GPU memory — training stores parameters, gradients, optimizer states, and activations; memory, not FLOPs, is often the first wall.
  • Forward/backward pass — forward computes activations and loss; backward walks layers in reverse to compute gradients.
  • All-reduce / collective — a coordinated operation where ranks combine values, such as summing gradients, and every rank receives the result.
  • Mixed precision — bf16/fp16 weights and activations reduce memory and bandwidth while selected states may remain fp32 for stability.
  • Adam optimizer states — Adam tracks momentum and variance for each parameter, commonly adding two fp32 tensors plus sometimes fp32 master weights.

1.Why scaling becomes a systems problem

The lecture begins with a practical observation: frontier models are trained at scales where a single accelerator is both too small and too slow. A trillion-parameter model, massive token stream, frequent checkpointing, and one-second-ish training-step targets put pressure on storage, networking, CPU scheduling, and GPU kernels at once.

1.1 The two hard constraints

There are two separate reasons to distribute training. First, the memory constraint: model states and activations may exceed one GPU. Second, the time constraint: processing the global batch sequentially would waste the parallel hardware. Good systems solve both without changing the mathematical training result.

Key equation — scaling efficiency

$$\text{speedup}(N)=\frac{T_1}{T_N}, \qquad \text{efficiency}(N)=\frac{T_1}{N\,T_N}$$

Here, \(T_1\) is one-GPU time, \(T_N\) is time on \(N\) GPUs. Linear scaling would mean efficiency near 1; communication and bubbles push it below 1.

✓ Check yourself — why is gradient accumulation alone not enough?

It reduces per-forward micro-batch size, but it still assumes the model states fit on one GPU and processes micro-batches sequentially. It helps batch size, not model-state memory or wall-clock scaling.

2.The memory problem

Before choosing a parallelism strategy, write down what consumes memory. During training, the GPU is not holding only weights. It also holds gradients, optimizer state, and saved activations needed by the backward pass. For long contexts, activations can dominate; for huge models, optimizer states can dominate.

2.1 What sits in GPU memory

A rough accounting is:

Key equation — training memory budget

$$M_{\text{train}} \approx P\,b_w + P\,b_g + P\,b_{\text{opt}} + M_{\text{act}} + M_{\text{workspace}}$$

\(P\) is parameter count. \(b_w\), \(b_g\), and \(b_{\text{opt}}\) are bytes per parameter for weights, gradients, and optimizer states. \(M_{\text{act}}\) is activation memory, which grows with batch, sequence length, hidden size, and layers.

  • Parameters: bf16 weights use about 2 bytes per parameter.
  • Gradients: often bf16/fp16 or fp32, depending on implementation.
  • Adam states: momentum and variance are often fp32, so 8 bytes/parameter; fp32 master weights can add 4 more bytes/parameter.
  • Activations: saved layer outputs for backward; checkpointing trades extra recompute for less activation storage.
✓ Check yourself — which term grows when sequence length grows?

Activation memory grows strongly with sequence length; attention-related activations can become especially expensive for long-context training. That is why sequence/context parallelism and activation checkpointing matter.

3.The parallelism dimensions

The lecture’s core taxonomy is that different axes split different things. The right recipe depends on what is scarce: memory, batch budget, intra-node bandwidth, inter-node bandwidth, or implementation simplicity.

3.1 Data, ZeRO/FSDP, tensor, and pipeline parallelism

  • Data parallelism (DDP): replicate the model, split the batch, compute gradients locally, then all-reduce gradients so every replica applies the same update.
  • ZeRO / FSDP: keep the data-parallel idea but shard optimizer states, gradients, and eventually parameters. ZeRO-3/FSDP all-gathers layer parameters just in time, computes, then frees them.
  • Tensor parallelism: split individual matrices or attention heads across GPUs. Megatron-style column/row splits reduce per-GPU compute and memory but require high-bandwidth collectives inside layers.
  • Pipeline parallelism: split layers into stages. Micro-batches flow through the stages to reduce the pipeline bubble, but activations must be saved or recomputed across many in-flight micro-batches.
Key equation — global batch and 3D mesh

$$B_{\text{global}} = B_{\text{micro}} \times G_{\text{accum}} \times D$$

\(D\) is the data-parallel degree. A common 3D layout has \(N = D \times T \times P\), where \(T\) is tensor-parallel degree and \(P\) is pipeline-parallel stages.

The “orthogonal axes” idea

If batch is split along the data axis, it is usually replicated along tensor groups. If weights are split along tensor groups, each data replica sees the same sharding pattern. This is why frameworks form process groups: a collective should run only along the axis that needs it.

3.2 Sequence/context and expert parallelism

  • Sequence parallelism: with tensor parallelism, replace certain all-reduces by reduce-scatter plus all-gather so activations can be sharded along the sequence dimension in non-TP regions.
  • Context parallelism / ring attention: split very long sequences across ranks; exchange keys and values so attention still covers the full context.
  • Expert parallelism: for mixture-of-experts models, shard experts across GPUs and route tokens with all-to-all dispatch/combine. Load balancing is essential; otherwise one expert rank becomes the busy kitchen station while others idle.
✓ Check yourself — why keep tensor parallelism inside a node when possible?

Tensor parallelism inserts collectives inside attention and MLP blocks, so it is communication-heavy and latency-sensitive. NVLink/NVSwitch within a node is much faster than inter-node InfiniBand/Ethernet, so intra-node TP hides more communication.

4.Keeping GPUs busy

Once the model fits, the bottleneck moves to utilization. Tazi repeatedly points at profiler timelines: the GPU compute stream should be full, the CPU should schedule kernels ahead of time, and communication should overlap with backward or forward compute where possible.

4.1 Overlap, checkpointing, precision, topology

  • Overlap communication with compute: DDP starts all-reduce on gradient buckets as soon as later-layer gradients are ready; FSDP prefetches the next module’s parameters while computing the current one.
  • Activation recomputation / checkpointing: save selected activations, recompute the rest in backward. This trades extra FLOPs for lower memory.
  • Mixed precision: bf16 reduces memory traffic and state size while retaining enough dynamic range for stable large-model training.
  • Pipeline scheduling: micro-batches fill the pipeline; 1F1B reduces activation pressure, while more advanced schedules such as DualPipe attack the bubble further.
  • Topology-aware placement: use NVLink/NVSwitch for chatty tensor parallel groups; reserve slower inter-node links for less frequent or better-overlapped collectives.
Key equation — MFU as the practical scorecard

$$\text{MFU}=\frac{\text{actual model FLOPs per second}}{\text{peak hardware FLOPs per second}}$$

MFU, model FLOPs utilization, asks: of the theoretical compute we bought, how much is doing model math rather than waiting on communication, CPU scheduling, data loading, or pipeline bubbles?

✓ Check yourself — what is the cost of activation checkpointing?

It lowers memory by not storing every intermediate activation, but backward must recompute some forward activations. You spend extra compute to buy memory headroom.

5.Worked example — memory + parallelism math

Now compute the small but honest case from the prompt: a 7B-parameter model trained in mixed precision. The point is not that 7B is frontier-scale; it is that even a familiar model size can exceed one 80 GB GPU once optimizer states are included.

Step-by-step with real numbers

Let \(P=7\times 10^9\) parameters.

$$\text{bf16 params}=P\times 2\text{ bytes}=14\text{ GB}$$

$$\text{bf16 grads}\approx P\times 2\text{ bytes}=14\text{ GB}$$

Adam often keeps fp32 momentum, fp32 variance, and fp32 master weights:

$$\text{Adam states}\approx P\times(4+4+4)\text{ bytes}=84\text{ GB}$$

So model states alone are roughly:

$$14+14+84=112\text{ GB}$$

That already exceeds an 80 GB GPU before activations, temporary workspaces, fragmentation, and communication buffers. A practical setup must shard states with ZeRO/FSDP, use tensor parallelism, reduce optimizer memory, or combine several methods.

✓ Check yourself — ZeRO-3 across 8 GPUs: what is the ideal per-GPU model-state memory?

ZeRO-3 shards parameters, gradients, and optimizer states, so the rough state memory is \(112/8=14\) GB per GPU, before live all-gathered layer windows, activations, buffers, and framework overhead. This is why FSDP makes the model feasible, but not “free.”

✓ Check yourself — if Adam states omit fp32 master weights, what changes?

The optimizer term becomes \(P\times(4+4)=56\) GB instead of 84 GB. The total state estimate becomes \(14+14+56=84\) GB, still above 80 GB before activations.

6.End-to-end trace — one training step across a cluster

Follow one micro-batch through a 3D-parallel cluster. The exact group sizes vary, but the handoffs are the recurring pattern.

StepWhat happensWhere/comms
0 · load micro-batchEach data-parallel rank receives its shard of the global batch; data workers try to stay ahead of GPU kernels.DP shard · no model comm yet
1 · forward beginsTensor-parallel ranks compute sharded attention/MLP matmuls; activations selected for checkpointing are saved sparsely.TP all-reduce / all-gather inside layers
2 · pipeline stagesStage 0 sends activations to stage 1, and micro-batches are interleaved to reduce the bubble.send/recv activations between PP stages
3 · loss and backwardBackward runs in reverse; recomputed activations are rebuilt where checkpointing was used.compute + recompute
4 · gradient synchronizationAs buckets become ready, gradients are all-reduced or reduce-scattered across data-parallel ranks.DDP all-reduce or ZeRO reduce-scatter
5 · optimizer stepEach ZeRO/FSDP rank updates only its local parameter/optimizer shards.sharded optimizer state
6 · prepare next stepNeeded parameters are all-gathered just in time for upcoming FSDP modules; the next micro-batch is already queued.FSDP prefetch + all-gather
Why communication—not FLOPs—usually caps scaling

Matrix multiplies are dense, predictable, and extremely fast on tensor cores. Cluster handoffs are less forgiving: they depend on topology, message size, synchronization, CPU launch overhead, and whether communication can be hidden. Once compute is split finely enough, each GPU’s math shrinks but the required coordination does not vanish.

✓ Check yourself — in the trace, where can idle time appear?

Data loading can starve step 0; tensor collectives can stall inside step 1; pipeline bubbles appear in step 2; un-overlapped gradient synchronization stalls step 4; slow all-gathers or CPU scheduling can delay step 6.

Glossary

TermPlain meaning
DDPDistributed data parallel: each GPU has a model replica, sees different data, then synchronizes gradients.
All-reduceA collective that combines values across ranks and returns the combined result to all ranks.
Reduce-scatterA collective that reduces values, then leaves each rank with only its shard of the result.
All-gatherA collective that gathers shards from ranks so each rank can see the assembled tensor.
ZeROZero Redundancy Optimizer: shards optimizer states, gradients, and parameters across data-parallel ranks.
FSDPFully Sharded Data Parallel: PyTorch’s ZeRO-3-style sharding of parameters, gradients, and optimizer state.
Tensor parallelismSplitting individual layer matrices or attention heads across GPUs.
Pipeline parallelismPlacing different model layers on different GPUs and streaming micro-batches through them.
Sequence/context parallelismSplitting long sequences across ranks to reduce activation/context memory.
Expert parallelismFor MoE models, splitting experts across GPUs and routing tokens to them.
Pipeline bubbleIdle time while pipeline stages wait for enough micro-batches or gradients to arrive.
MFUModel FLOPs utilization: actual model computation divided by peak available hardware computation.
