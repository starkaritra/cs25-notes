A Recipe for Training Helpful Chatbots

Nazneen Rajani explains the practical alignment kitchen: start with a pretrained LM, teach it to follow instructions, collect human or AI preferences, then optimize toward helpful, honest behavior without letting the model drift into reward-gaming weirdness.

🎓 Nazneen Rajani / Hugging Face ⏱ ~50 min ▶ Watch
Why this matters

A raw pretrained language model is very good at the game it was trained to play: predict plausible next text. But a chatbot user is not asking for “whatever text tends to come next on the internet.” They want an answer that follows the instruction, stays factual, avoids harm, and admits uncertainty. That is a different target.

This lecture complements Note 12 on alignment. Note 12 gives the strategic framing; this note gives the hands-on recipe Hugging Face's H4 team used when trying to recreate the “secret sauce” of aligned chat models in the open.

Running analogy — cooking an assistant

Think of a helpful chatbot as a dish, not a raw ingredient. Pretraining grows the staple crop. SFT teaches the recipe card. Preference data is the tasting panel. Reward modeling learns the panel's palate. PPO with a KL leash, or DPO, adjusts the seasoning while preventing the chef from dumping in sugar because the taste-test metric liked one sweet bite.

Prerequisites — 30-second refresher

You only need a few concepts active before the pipeline starts.

  • Language modeling — pretraining minimizes next-token surprise, e.g. \(-\sum_t \log p_\theta(x_t\mid x_{<t})\); this gives capability, not an assistant persona.
  • Instruction tuning / SFT — fine-tune on pairs \((\text{instruction},\text{good response})\) so the model imitates demonstrated assistant behavior.
  • Reward and policy in RL — the policy \(\pi(y\mid x)\) produces a response \(y\) to prompt \(x\); a reward \(r(x,y)\) scores how desirable that response is.
  • Logistic sigmoid — \(\sigma(z)=1/(1+e^{-z})\) turns a score difference into a probability; preference models use it to say “winner should score above loser.”
  • KL divergence — \(D_{\mathrm{KL}}(p\|q)\) measures distribution drift; RLHF uses it as a leash from the RL-updated model back to the supervised assistant.

1.The assistant gap: prediction is not helpfulness

Imagine asking, “Can you help me debug this Python error?” A base LM may continue with a forum-like thread, hallucinate package names, or answer in a style that looks plausible but does not solve the problem. The model is not being stubborn; its original objective never said “be the user's assistant.”

1.1 H4 and the HHHH target

Rajani describes Hugging Face's H4 effort: build an open recipe for a chatbot that is helpful, harmless, honest, and Huggy. The team did not try to redo pretraining. They started with open pretrained models and focused on the alignment stages that turn a capable text predictor into something chat-shaped.

Her practical split is useful: SFT mostly teaches the model to be helpful and conversational; preference learning plus RLHF adds stronger pressure toward desirable behavior such as truthfulness and harmlessness. In real systems these blend, but this mental split helps you debug the recipe.

1.2 From web text to chat turns

A chat model is trained on structured turns, not just loose documents. A dataset example might contain a system message, a user instruction, optional input context, and an assistant response. The chat template tells the tokenizer and model where those turns begin and end.

Key equation — base LM objective versus assistant target

$$ \underbrace{\min_\theta\;\mathbb{E}_{x\sim\mathcal{D}_{\text{text}}}\left[-\log p_\theta(x_t\mid x_{<t})\right]}_{\text{predict likely text}} \quad\neq\quad \underbrace{\max_\pi\;\mathbb{E}_{x\sim\mathcal{D}_{\text{users}},\,y\sim\pi(\cdot\mid x)}\left[U_{\text{human}}(x,y)\right]}_{\text{help the user}} $$

The left side learns internet regularities. The right side asks for human utility: following the instruction, being useful, staying truthful, and refusing appropriately.

Common misconception

“If the base model is strong enough, it is already aligned.” Scale can improve knowledge and fluency, but it does not automatically prioritize the user's intent. Rajani's whole recipe exists because capability and helpful assistant behavior are separable ingredients.

✓ Check yourself — which part of a raw LM's objective says “be concise and truthful”?

None directly. The model may learn patterns of concise or truthful text when those patterns appear in pretraining, but the loss only rewards predicting observed tokens, not satisfying the user's actual preference.

2.The training recipe: from base LM to aligned chat

The canonical picture Rajani starts from is InstructGPT: demonstrations first, preferences second, reward-guided optimization third. Hugging Face's H4 work keeps this skeleton, then asks very practical questions: which data, how much, what task mix, human or synthetic, and how do we evaluate?

2.1 SFT / instruction tuning: demonstrations teach the format

Start with examples where the input and output are both supplied by humans or high-quality synthetic processes. Each item says, “when the user asks this, a good assistant answer looks like that.” In the transcript, Rajani highlights task categories such as generation, open-ended Q&A, brainstorming, classification, summarization, and code-related help.

Key equation — supervised fine-tuning

$$ \mathcal{L}_{\mathrm{SFT}}(\theta) =-\mathbb{E}_{(x,y^*)\sim\mathcal{D}_{\mathrm{demo}}}\log \pi_\theta(y^*\mid x) $$

\(x\) is the prompt, \(y^*\) is the demonstrated assistant response, and \(\pi_\theta\) is the policy being trained. In the cooking analogy, this is the chef copying good recipes before inventing variations.

✓ Check yourself — why do chat templates matter during SFT?

They make the training and inference format match. If the model sees explicit user/assistant/system turn markers during fine-tuning, it learns when to answer, when to condition on system instructions, and where the assistant response should begin.

2.2 Preferences and reward models: learn the tasting panel

For preference data, humans write or select prompts; models generate multiple candidate responses; annotators rank, rate, or choose the better response. Rajani emphasizes that this is often easier than writing the perfect answer, but still noisy: if two bad model outputs are nearly tied, annotators may disagree or break ties arbitrarily.

Key equation — Bradley–Terry reward-model loss

$$ \mathcal{L}_{\mathrm{RM}}=-\log\sigma\big(r(x,y_w)-r(x,y_l)\big) $$

The reward model \(r(x,y)\) gives a scalar score to a prompt-response pair. \(y_w\) is the preferred winner, \(y_l\) is the less-preferred loser, and \(\sigma\) turns the score gap into “probability the winner is preferred.”

2.3 PPO with KL, or DPO without a separate reward model

Classical RLHF trains a reward model, then uses PPO to update the assistant policy toward high-reward responses. But if the policy can exploit blind spots in the reward model, it may drift away from normal helpful language. The KL penalty anchors it to the SFT policy.

Key equation — RLHF objective with KL leash

$$ \max_\pi\;\mathbb{E}_{x,\,y\sim\pi(\cdot\mid x)}\left[ r_\phi(x,y)-\beta D_{\mathrm{KL}}\!\left(\pi(\cdot\mid x)\,\|\,\pi_{\mathrm{SFT}}(\cdot\mid x)\right) \right] $$

The reward term says “make the tasting panel happy.” The KL term says “do not leave the recipe book entirely.” \(\beta\) controls how tight the leash is.

DPO is the modern shortcut Rajani discusses in the Zephyr work: it uses preference pairs to optimize the policy directly, without first training a separate reward model and running an RL loop. It is still preference optimization, but the implementation is simpler and often more stable.

✓ Check yourself — what stage disappears in DPO compared with PPO-style RLHF?

The explicit learned reward model plus PPO rollout loop. DPO trains the policy directly from chosen/rejected preference pairs, usually relative to a reference policy.

3.Data is the crux, not an afterthought

A recipe is only as good as its ingredients. Much of the lecture is not about a new loss function; it is about the unglamorous details that decide whether the loss sees useful signal: task mix, length distribution, dialogue turns, annotator instructions, and whether synthetic data helps or contaminates evaluation.

3.1 Human and synthetic demonstrations

Rajani maps instruction datasets along a spectrum. On the human-written side are efforts such as Surge-collected H4 data, OpenAssistant, and Dolly. On the synthetic side are pipelines such as Self-Instruct, UltraChat, and role-play datasets like CAMEL, where stronger LMs generate tasks, answers, or whole conversations.

  • Human demonstrations can be high quality, factual, and tailored, but they are expensive and slower to collect. H4 spent heavily on roughly 10k SFT pairs plus 20k preference dialogues.
  • Synthetic / distilled data scales fast. UltraChat and UltraFeedback enabled Zephyr-7B to become a strong open chat model, but the model may inherit the style and biases of the teacher and evaluator.
  • Diversity matters: generation, brainstorming, coding, factual QA, summarization, and chat stress different abilities. Rajani notes that factual tasks are especially useful when collecting preferences because correctness is less ambiguous.
Ingredient quality beats ingredient pile size

Several results in the lecture echo LIMA and Self-Instruct: after a few thousand high-quality instruction examples, returns can saturate. More examples are not automatically better if they repeat the same short, easy, or noisy patterns.

3.2 Preference annotation reliability

Preference data sounds simple: show two responses and ask which is better. In practice, Rajani's team found low agreement when model outputs were both weak. The annotation template also matters: Anthropic-style 1–8 pairwise rankings, LLaMA-2-style “which is better and by how much,” or individual 1–10 scoring all induce slightly different noise.

Common misconception

“Preferences are objective labels like ImageNet classes.” Some prompts have clear truth conditions; many do not. A verbose answer, a concise answer, and a safety-conscious answer may trade off differently depending on the user's real need.

✓ Check yourself — why did H4 emphasize factual tasks in preference collection?

Because factual prompts make it easier to distinguish truthful/helpful from untruthful/unhelpful. Open-ended generation or brainstorming can have many acceptable answers, so preference labels become noisier.

4.Worked micro-example — one reward-model preference by hand

Numbers make the Bradley–Terry loss less mysterious. Suppose the prompt is: “Explain why the sky is blue in two sentences.” A human prefers response \(y_w\) over response \(y_l\). The reward model currently assigns:

Step-by-step with real numbers

$$ r_w=r(x,y_w)=2.2,\qquad r_l=r(x,y_l)=0.7 $$

First compute the score gap:

$$ \Delta=r_w-r_l=2.2-0.7=1.5 $$

Turn the gap into the model's probability that the winner is preferred:

$$ \sigma(1.5)=\frac{1}{1+e^{-1.5}}\approx\frac{1}{1+0.2231}=0.8176 $$

Now compute the loss:

$$ \mathcal{L}=-\log(0.8176)\approx 0.2014 $$

The loss is already fairly small because the reward model scores the winner above the loser. The gradient-sign intuition is simple: if the label says \(y_w\) won, training pushes \(r_w\) upward and \(r_l\) downward, especially when \(\Delta\) is too small or negative.

✓ Check yourself — what if both responses had equal scores?

If \(r_w=r_l\), then \(\Delta=0\), \(\sigma(0)=0.5\), and \(\mathcal{L}=-\log(0.5)\approx0.693\). The model is saying “coin flip,” so the preference label gives a strong push: raise the winner's score and lower the loser's score.

Why the KL penalty matters

The reward model is only an approximation of human taste. Without the KL anchor, PPO can discover strange text that scores well for the reward model but no longer looks like normal helpful assistant behavior. KL says: improve the seasoning, but do not mutate the dish into something the original recipe would never serve.

✓ Check yourself — at which stage do humans usually compare multiple model answers?

After SFT, when the model can already produce chat-like candidates. Humans compare or rank sampled responses, creating preference data for a reward model or DPO.

6.Evaluation and failure modes: measuring a chatbot is also alignment

Rajani's last arc is a warning: once you train chatbots, you still need to know whether they improved. Old NLP benchmarks test pieces of capability; chat assistants require human preference, instruction following, harmlessness, and robustness under adversarial prompting.

6.1 Human eval, LLM-as-judge, MT-Bench, Arena, Elo

Human evaluation remains the closest target, but it is slow and expensive. Public leaderboards therefore use pairwise wins, Elo-style ratings, GPT-4 or Claude as judges, MT-Bench multi-turn scoring, AlpacaEval, and Chatbot Arena crowdsourced battles. These are useful thermometers, not laws of nature.

Key equation — Elo-style pairwise intuition

$$ P(A\text{ beats }B)=\frac{1}{1+10^{(R_B-R_A)/400}} $$

\(R_A\) and \(R_B\) are model ratings inferred from pairwise comparisons. If many users or judges prefer model A over model B, A's rating rises. The catch: judge biases become rating biases.

6.2 What can go wrong

  • Reward hacking — the policy finds outputs that exploit the reward model rather than actually helping the user.
  • Sycophancy — the model flatters or agrees with the user instead of correcting false premises.
  • Over-refusal — harmless requests get blocked because the safety signal is too blunt.
  • Verbosity bias — judges, especially LLM judges, may prefer longer list-like answers even when concise answers are better.
  • Position bias — Rajani reports GPT-4 showing a left-position preference in pairwise judging; telling it about the bias can even flip the bias.
  • Evaluation doping — a judge may favor models trained on data generated by that same judge model, making synthetic-data wins look cleaner than they are.
Common misconception

“GPT-4 judging is a free replacement for humans.” It is scalable and often useful, but Rajani shows concrete quirks: positional bias, preference for GPT-4-like data, stronger correlation on creative tasks than on some low-entropy math/coding/reasoning tasks, and sensitivity to answer style.

✓ Check yourself — how can you reduce position bias in pairwise LLM judging?

Randomize or balance which model appears on the left and right, evaluate both orders when possible, and report sensitivity. Do not assume a single fixed order is neutral.

Glossary

TermPlain meaning
Base LMA pretrained language model optimized to predict text, before chat-specific alignment.
SFT / instruction tuningSupervised fine-tuning on instruction-response demonstrations so the model imitates good assistant answers.
Chat templateThe exact token format for system, user, and assistant turns.
Preference dataExamples where humans or AI judges choose or rank candidate responses to the same prompt.
Reward modelA learned scorer \(r(x,y)\) that predicts which response people prefer.
Bradley–Terry lossA pairwise loss that makes a preferred response score higher than a rejected response.
RLHFReinforcement learning from human feedback: optimize a policy against a reward model trained from preferences.
PPOA policy-gradient RL algorithm often used to update the assistant policy during RLHF.
KL penaltyA drift penalty that keeps the RL-updated policy close to a reference model, usually the SFT model.
DPODirect Preference Optimization: train directly from chosen/rejected pairs without a separate reward-model RL loop.
Red-teamingAdversarial testing that probes whether the model reveals harmful, deceptive, private, or otherwise unsafe behavior.
EloA rating system inferred from pairwise wins and losses, adapted from chess to chatbot comparisons.
