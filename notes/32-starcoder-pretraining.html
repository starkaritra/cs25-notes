Behind the Scenes of LLM Pre-training: The StarCoder Use Case

Loubna Ben Allal uses StarCoder and BigCode to show that strong LLMs are not built by architecture alone: the hidden craft is collecting, governing, filtering, tokenizing, training, and evaluating the data pipeline end to end.

🎓 Loubna Ben Allal · Hugging Face / BigCode ⏱ ~50 min ▶ Watch
Why this matters

Most public conversations about LLMs point at the transformer block, parameter count, or GPU cluster. Loubna's central message is sharper: once many teams use similar decoder-only transformers, data quality becomes the lever that separates a useful model from an expensive memorizer. StarCoder is valuable because BigCode documented the unglamorous parts: license filtering, near-deduplication, PII redaction, benchmark decontamination, token formatting, evaluation, and governance.

Running analogy

Think of pre-training a code LLM like preparing a city-scale public library for an apprentice programmer. The transformer is the reading brain, but the library staff decides what enters the shelves, removes duplicate photocopies, redacts private notes, labels every aisle, and tests whether the apprentice can solve new exercises rather than recite old answer keys.

Prerequisites — 30-second refresher

Skim these before the tour. Each one will reappear in the StarCoder pipeline.

  • Language-model pre-training — train on raw token sequences by predicting missing or next tokens; here, the raw text is mostly source code.
  • Next-token objective — maximize \\(p_\\theta(x_t \mid x_{<t})\\), so the model learns regularities that make the next token likely.
  • Tokenization — convert characters into model tokens; code cares about indentation, identifiers, punctuation, and file metadata.
  • Deduplication — remove exact or near copies so training is not dominated by repeated files and evaluation leakage.
  • Decoder-only transformer — a causal transformer that reads left-to-right; StarCoder adds tricks so it can also learn infilling.
  • HumanEval / pass@k — sample \\(k\\) candidate programs, run unit tests, and ask whether at least one passes.

1.The real bottleneck is the data pipeline

Suppose two teams both train a transformer-like code model. Team A scrapes everything it can find; Team B starts with public code, removes legally risky sources, deletes near-duplicates, redacts secrets, filters junk, and removes benchmark lookalikes. Loubna's argument is that Team B often wins even before changing the architecture, because its model spends gradient updates on useful programming patterns rather than repeated boilerplate, passwords, generated trash, or evaluation answers.

1.1 The Stack: code, but governable

BigCode's training story begins with The Stack, a large corpus of permissively licensed GitHub code. The lecture describes cloning public repositories, filtering file extensions, keeping permissive licenses such as MIT or Apache-2.0, deduplicating, and supporting an opt-out tool so repository owners could inspect and request removal. The Stack v2 later used the Software Heritage archive and grew to hundreds of billions of training tokens.

Key equation — training data as weighted evidence

$$ \mathcal{L}(\theta) = -\sum_{d \in \mathcal{D}_{\text{clean}}}\sum_{t=1}^{|d|} \log p_\theta(x_t \mid x_{<t}) $$

Here \\(d\\) is one cleaned document, \\(x_t\\) is token \\(t\\), and \\(\mathcal{D}_{\text{clean}}\\) is not "the internet"; it is the curated corpus after license, deduplication, PII, quality, and decontamination filters. Change \\(\mathcal{D}_{\text{clean}}\\), and every gradient step changes.

1.2 Cleaning beats wishful scale

The lecture makes deduplication concrete: removing similar files can produce a large benchmark bump because the model no longer over-spends compute on repeated templates. PII redaction is another non-negotiable stage. BigCode annotated examples of names, emails, keys, and passwords, trained a StarPII named-entity model, and ran it over the StarCoder training data. Decontamination then removes benchmark and test-set content so evaluation is not inflated.

Common misconception

"More raw tokens always means a better model." Not if many tokens are duplicate files, generated sludge, leaked secrets, or benchmark answers. In the library analogy, adding ten million photocopies of the same manual does not make the apprentice ten million times wiser; it crowds out shelf space and practice time.

✓ Check yourself — which filter protects which stakeholder?

License filtering protects creators and downstream users from unclear rights; PII redaction protects people whose secrets or personal identifiers appeared in code; deduplication protects training efficiency and benchmark validity; opt-out protects contributor agency; benchmark decontamination protects scientific claims.

2.Tokenizing code as code

A text tokenizer that works for prose can still waste capacity on code. In code, a missing space can change a block, an identifier like parse_http_response carries compressed meaning, punctuation is syntax, and filenames can tell the model which language it should write. StarCoder used a BPE-style tokenizer trained on its code mixture, plus formatting tokens that expose metadata such as repository name, filename, and stars.

2.1 Identifiers, whitespace, and metadata

In the library analogy, tokenization is the cataloging system. If the catalog splits useful identifiers poorly or hides the filename, the apprentice sees a stream of fragments without aisle labels. BigCode added special formatting so an IDE use case can say, in effect, "this is src/model.py from this repository" before the code begins.

Key equation — sequence length after tokenization

$$ n = n_{\text{metadata}} + n_{\text{code}} \leq 8192 $$

For StarCoder, \\(n\\) is the number of tokens placed in the context window, \\(n_{\text{metadata}}\\) covers tokens such as filename or repository labels, and \\(n_{\text{code}}\\) covers the code itself. The engineering question is how much useful project context fits before the model must predict.

✓ Check yourself — why might a filename token help?

utils/math_ops.py, package.json, and Dockerfile imply different languages, libraries, and conventions. A filename token gives the model a cheap prior before it reads the body of the file.

3.Training StarCoder

After curation and tokenization, the model choice is intentionally familiar: StarCoderBase is a decoder-only transformer with roughly 15B parameters, trained on about 1T code tokens from The Stack, then StarCoder is further trained on Python. The interesting parts for a code assistant are not a magical new block; they are long context, cheap inference, and infilling.

3.1 Decoder-only, MQA, and long context

A code assistant inside VS Code may need the current file plus neighboring files. That makes an 8k-token context useful. But serving long contexts is expensive, so StarCoder uses Multi-Query Attention (MQA): many query heads, but shared key/value heads. The model can keep expressive query projections while reducing key-value cache memory during inference.

Key equation — attention and the MQA memory idea

$$ \operatorname{Attn}(Q,K,V)=\operatorname{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right)V $$

Standard multi-head attention stores separate \\(K,V\\) caches per head. With MQA, \\(Q_h\\) can vary by head \\(h\\), while \\(K\\) and \\(V\\) are shared, reducing inference cache from roughly \\(H \cdot L \cdot d_k\\) to \\(L \cdot d_k\\) per layer for keys and values, where \\(H\\) is head count and \\(L\\) is context length.

3.2 Fill-in-the-Middle as an IDE skill

Plain causal training teaches "given the left side, write what comes next." IDE users often need something harder: "given the code before and after a cursor gap, fill the missing function body." Fill-in-the-Middle (FIM) trains that use case by splitting a document into prefix \\(P\\), middle \\(M\\), and suffix \\(S\\), then rearranging tokens so the model sees both sides before generating the missing middle.

Key equation — FIM rearrangement

$$ [P, M, S] \quad\longrightarrow\quad [\texttt{<PRE>}, P, \texttt{<SUF>}, S, \texttt{<MID>}, M] $$

\\(P\\) is the code before the gap, \\(S\\) is the code after the gap, and \\(M\\) is the missing span. During training, the model predicts the tokens of \\(M\\) after it has read both \\(P\\) and \\(S\\).

✓ Check yourself — why is FIM especially natural for code?

Code is full of constraints from both sides: a function body must match its signature above and its callers, return type, indentation, imports, and tests below. FIM lets the model condition on those right-side constraints instead of pretending completion always happens at the end of a file.

4.Evaluation and contamination

For code models, evaluation can be executable: ask the model to complete a function, then run unit tests. The lecture highlights HumanEval as a common benchmark, MBPP as another Python benchmark, and MultiPL-E as a multilingual translation of HumanEval-style problems. This is stronger than judging fluency, but it is also fragile: if benchmark solutions or near copies are in the training data, the score becomes a memory test.

4.1 pass@k, HumanEval, MBPP, MultiPL-E

The basic metric samples \\(k\\) candidate programs for each problem. If at least one passes the tests, the problem counts as solved for pass@\\(k\\). pass@1 is the strict single-shot version; larger \\(k\\) measures whether the model can produce a correct answer somewhere in a batch of attempts.

Key equation — pass@k intuition

$$ \operatorname{pass@}k \approx \frac{\#\{\text{problems with at least one passing sample among }k\}}{\#\{\text{problems}\}} $$

The exact unbiased estimator adjusts for the number of generated samples, but the intuition is this: generate candidates, run tests, and count problems solved at least once.

Common misconception

"A higher HumanEval score always means deeper programming ability." Maybe, but maybe the training set contained similar exercises, solutions, or synthetic tasks that look too much like HumanEval. That is why BigCode's data decontamination stage matters.

✓ Check yourself — why is executable evaluation still not enough?

Unit tests can miss edge cases, benchmarks can be contaminated, and a model may overfit to benchmark style. Executable tests are valuable, but the pipeline still needs decontamination, multilingual tests, and qualitative inspection.

5.Openness is part of the method

BigCode's contribution is not only a model; it is a reproducibility package. The StarCoder release includes model weights, data-processing code, evaluation tools, the preprocessed training data, PII resources, attribution tooling, an opt-out process, and an OpenRAIL-style responsible-AI license. That matters because Loubna frames open LLM progress as the community learning the missing details that closed releases often omit.

5.1 Weights, data, process, license

The governance story is part scientific and part social. Openness lets researchers inspect, reproduce, fine-tune, and audit. Responsible release constrains harmful uses, documents data choices, and gives contributors mechanisms such as opt-out. In the library analogy, BigCode is not merely lending out the apprentice; it is publishing the cataloging rules and the library policy.

✓ Check yourself — what does openness add beyond downloading weights?

Weights enable use; process enables trust and reproduction. Data cards, preprocessing code, opt-out tools, PII redaction, checkpoints, and evaluation harnesses let others inspect what happened and build on it responsibly.

6.Worked example — Fill-in-the-Middle by hand

Take a tiny Python file and cut out the part an IDE user wants filled. The original code has a prefix \\(P\\), a missing middle \\(M\\), and a suffix \\(S\\).

# Original file
P: def add_tax(price, rate):
M:     return price * (1 + rate)
S:
print(add_tax(100, 0.07))
Step-by-step token rearrangement

Define the three spans:

$$ P=\texttt{"def add\_tax(price, rate):\\n"},\quad M=\texttt{" return price * (1 + rate)\\n"},\quad S=\texttt{"print(add\_tax(100, 0.07))\\n"} $$

FIM rewrites the training sequence as:

$$ \texttt{<PRE>}\;P\;\texttt{<SUF>}\;S\;\texttt{<MID>}\;M $$

Concretely:

<PRE>
def add_tax(price, rate):
<SUF>
print(add_tax(100, 0.07))
<MID>
    return price * (1 + rate)

At inference time, the IDE can provide the same left and right context up to <MID>. The model's next-token predictions are then trained to produce the missing body, not merely the next line after the prefix.

✓ Check yourself — change the suffix, predict the effect

If the suffix became assert add_tax(100, 0.07) == 107, a good FIM-trained model should be pushed toward a numeric return expression compatible with that assertion. The right context is evidence, not decoration.

7.End-to-end trace — one code file becomes StarCoder training signal

Follow one permissively licensed Python file from public repository to model behavior. The sizes are scale anchors from the lecture and BigCode releases, not exact counts for this one file.

StepWhat happensData/size
0 · raw GitHubPublic repositories are collected or accessed through Software Heritage.TBs to 100+ TB raw source
1 · license filterKeep repositories/files with permissive licenses; remove unclear or excluded licenses.The Stack v1: 6TB permissive release
2 · dedupRemove exact and near-duplicate files so repeated boilerplate does not dominate.v1 near-dedup: ~3TB
3 · PII scrubDetect and redact names, emails, keys, passwords, and other sensitive spans.StarPII over StarCoder data
4 · quality filterDrop low-signal files, unwanted extensions, benchmark/test leakage, and noisy samples.cleaner training subset
5 · tokenizeApply code-trained tokenizer and metadata tokens such as repository and filename.tokens, not raw characters

What the trace reveals

The model never trains on "GitHub" in the abstract. It trains on the distribution left after dozens of governance and quality decisions. That is why data curation dominates final model quality: each filter changes what the apprentice practices, what it memorizes, what it is allowed to reveal, and what claims we can make about its benchmarks.

Glossary

TermPlain meaning
The StackBigCode's large dataset of permissively licensed source code used for code-LLM pre-training.
Permissive licenseA software license such as MIT or Apache-2.0 that generally allows broad reuse under stated conditions.
Near-deduplicationRemoving files that are not byte-identical but are similar enough to act like duplicates.
PII redactionDetecting and removing personal or sensitive information such as names, emails, passwords, and keys.
DecontaminationRemoving evaluation benchmark examples or lookalikes from training data to avoid inflated scores.
BPE tokenizerA tokenizer that learns frequent byte/subword merges; useful for splitting identifiers and code symbols into reusable pieces.
Decoder-only transformerA causal transformer architecture that predicts future tokens from previous context.
MQAMulti-Query Attention: multiple query heads share key/value projections to reduce inference cache cost.
FIMFill-in-the-Middle: rearrange prefix, suffix, and middle spans so the model learns to fill a gap using both sides.
pass@kA code-evaluation metric asking whether at least one of \\(k\\) sampled programs passes the tests.
OpenRAILResponsible-AI license family used to release models with open access plus usage restrictions.
