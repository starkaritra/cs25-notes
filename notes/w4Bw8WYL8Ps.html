Decision Transformer: Reinforcement Learning via Sequence Modeling

A portfolio-grade map of the lecture's central move: turn offline reinforcement learning into GPT-style action prediction, conditioned on the return you want.

🎓 Speaker: Aditya Grover · Chen et al. (2021) ⏱ ~50 min ▶ Watch on YouTube
Why this lecture matters

Transformers scaled cleanly in language and vision: bigger data, bigger models, smoother losses. Deep reinforcement learning has not felt like that. Classic RL often depends on TD-learning, Bellman backups, bootstrapping, actor-critic machinery, exploration heuristics, and high-variance training. The lecture asks a radical question: can we skip most of that and treat RL as sequence modeling?

Running analogy for this note

Imagine a coach with a library of game films. Instead of deriving equations for the perfect strategy, the coach says: "Show me every play from games where the team scored 90 points; now learn what a 90-point team tends to do next." Decision Transformer is that coach with a transformer: study trajectories, condition on a desired score, predict the next action.

Prerequisites — 30-second refresher

Skim these; dive into any that feel shaky.

  • State / action / reward — at each step the agent sees state \(s_t\), takes action \(a_t\), gets reward \(r_t\).
  • Return — the sum of future rewards; the thing RL tries to maximize.
  • Causal transformer — a GPT-style model that predicts each token from earlier ones only (masked).
  • Token — here, one element of the trajectory sequence (a return-to-go, a state, or an action).

1.Quick RL primer for NLP/CV readers

Start with a concrete game. In Atari Pong, the state is the screen, the action might be moving the paddle up or down, and the reward arrives when someone scores. The hard part is that an action now may only matter many frames later.

1.1 Agent, environment, policy

Reinforcement learning studies an agent interacting with an environment over time:

  • At time step t, the agent observes a state \(s_t\).
  • It samples or chooses an action \(a_t\) from a policy \(\pi(a_t\mid s_t)\).
  • The environment returns a reward \(r_t\) and the next state \(s_{t+1}\).
  • The goal is to maximize expected return: the sum of future rewards.
Concrete before general

If the paddle moves up and the ball is returned five frames later, the reward is not attached neatly to the single frame where the action was chosen. RL is hard because good decisions and evidence of goodness are separated in time.

✓ Check yourself — what is the policy?

The policy \(\pi\) is the agent's action rule: given what it has observed (often \(s_t\), sometimes a longer history), it outputs an action or a distribution over actions.

1.2 Return-to-go: the number DT will condition on

The lecture uses return-to-go, not immediate reward, as a token. For an episode of length \(T\):

Key equation — return-to-go

$$\hat R_t = \sum_{t' = t}^{T} r_{t'}$$

Read \(\hat R_t\) as: "from now until the end, how much reward remains to be collected?" If a trajectory earns rewards \([0,0,1,0,2]\), then at the start \(\hat R_1=3\); after the reward of 1 is collected, the remaining target drops.

Do not confuse these

The reward \(r_t\) is local: "what happened now?" The return-to-go \(\hat R_t\) is global: "how good is the rest of this trajectory?" Decision Transformer uses the global number as a conditioning signal.

✓ Check yourself — if future rewards are [2, 0, 5], what is the current return-to-go?

\(2+0+5=7\). With a discount factor \(\gamma\), one could use a discounted return, but the core Decision Transformer experiments commonly use undiscounted return-to-go.

2.Offline RL: learn from fixed trajectories

Decision Transformer is introduced in the offline RL setting. You are handed a static dataset of trajectories:

$$\mathcal{D}=\{(s_1,a_1,r_1,s_2,a_2,r_2,\ldots)\}_{\text{many episodes}}$$

You may train on this log, but you may not keep interacting with the environment while learning.

Why offline RL is valuable

In real systems — robots, hospitals, finance, autonomous vehicles — trial-and-error exploration can be unsafe, expensive, or unethical. Offline RL says: use logged behavior first. That makes the setup more realistic for many high-stakes domains and a natural place to test whether transformer-style supervised learning can help.

✓ Check yourself — why is offline RL harder than ordinary supervised learning?

The dataset contains actions from some behavior policy, not necessarily the best policy. At test time, if the learned policy visits states poorly covered by the log, its predictions can be unreliable.

3.The Decision Transformer core idea

The core move of Chen et al. (2021) is almost disarmingly simple: represent a trajectory as a sequence of tokens and train a causal transformer to predict the action token. No Bellman backup is required in the basic algorithm.

3.1 A trajectory becomes a language-model sequence

Instead of feeding only states, Decision Transformer interleaves return-to-go, state, action triplets:

$$\left(\hat R_1, s_1, a_1,\; \hat R_2, s_2, a_2,\; \hat R_3, s_3, a_3,\; \ldots\right).$$

At time \(t\), the model is asked to predict \(a_t\) from the tokens before it: desired return so far, current state, and previous history.

What changed conceptually?

Classic RL asks: "Which action maximizes value under the Bellman equation?" Decision Transformer asks: "In trajectories that still had return \(\hat R_t\) left to earn and were in state \(s_t\), what action usually came next?" That is a supervised sequence-modeling question.

3.2 Training: GPT-style causal prediction, supervised loss

Each token type gets an embedding. A timestep embedding is added, and a GPT-like transformer processes the sequence causally: each position can only attend to earlier positions. The main prediction head is attached to state-token positions to output the next action.

  • Discrete actions (Atari): train with cross-entropy.
  • Continuous actions (MuJoCo / D4RL): train with mean-squared error or a continuous action likelihood.
  • Context length \(K\): the model attends over a finite window of previous triplets, just as language models use a context window.
Training objective — action imitation with return conditioning

The loss is ordinary supervised learning. The RL flavor enters through how the sequence is constructed and what return we condition on.

✓ Check yourself — where did the Bellman backup go?

It is not used in the basic Decision Transformer objective. The model does not learn a value function by bootstrapping from its own estimates; it learns to predict dataset actions conditioned on return-to-go.

4.Return conditioning: "upside-down" RL

The key twist happens at test time. There is no completed trajectory yet, so we choose an initial desired return-to-go — often near or slightly above the best return in the dataset — and use it as a prompt.

  1. Prompt the model with a target \(\hat R_1\) and the initial state \(s_1\).
  2. Generate an action \(a_1\).
  3. The environment returns reward \(r_1\) and next state \(s_2\).
  4. Update the remaining target: \(\hat R_2 = \hat R_1 - r_1\).
  5. Append the new tokens and repeat autoregressively.
The memorable analogy

Reward becomes an input, not merely an objective signal. You tell the model: "play like a 90-point player." If it has seen enough 90-point-ish behavior, the transformer can imitate the action patterns associated with that level of performance.

This is not magic wish fulfillment

Typing an impossible target return does not create new physics or new data coverage. Empirically, very high targets often saturate: the model behaves like the best behavior it knows rather than improving forever.

✓ Check yourself — how is DT like prompting a language model?

A language model continues text conditioned on a prompt. DT continues a trajectory conditioned on a prompt containing desired return and current state. The "next token" it emits is an action.

5.Why credit assignment becomes easier

Classic TD methods pass information backward through bootstrapped value estimates: reward informs \(V(s_T)\), which informs \(V(s_{T-1})\), and so on. That can be unstable and slow, especially when rewards are sparse or delayed.

The key-to-door story

The lecture highlights a sparse-reward environment: the agent must pick up a key early, pass through an empty room, then open a door much later. A TD learner must propagate credit back through many transitions. A transformer sees the whole relevant sequence segment and can learn that "picked up key" is predictive of eventual success.

Careful wording

Decision Transformer does not universally "solve" credit assignment. It makes one important version easier when the useful evidence lies inside the context window and the offline dataset contains examples that reveal the connection.

✓ Check yourself — why does sparse reward hurt TD methods?

If most transitions have reward 0, the learning signal must be bootstrapped backward across many steps. Errors compound, and many updates may be needed before the early action receives useful credit.

6.Results and benchmarks

The headline result is not "DT instantly replaces all offline RL." It is stronger and more interesting: a simple causal transformer with a supervised loss is competitive with specialized offline-RL algorithms across standard domains, and it looks especially promising where long-horizon credit assignment matters.

The reported pattern is also a scaling story: as the transformer context and capacity increase, DT can use more trajectory history and often improves, echoing the model-size scaling intuition that made transformers powerful in NLP and vision.

Benchmark familyWhat it testsLecture takeaway
AtariDiscrete actions, image-like game observations, replay-buffer datasets.DT is competitive with strong offline RL baselines and can work with relatively small replay-buffer fractions.

Why the scaling point matters

The lecture repeatedly contrasts RL with language modeling: language models scale to billions of parameters with stable supervised losses, while deep RL policies often remain smaller and more brittle. DT is exciting because it imports the training recipe that already scales.

Cousin model: Trajectory Transformer

Trajectory Transformer appeared around the same time. It also treats RL trajectories as sequences, but more explicitly models states, actions, and rewards for planning. Decision Transformer is deliberately simpler: condition on return-to-go and directly predict actions.

✓ Check yourself — why is percent behavioral cloning a serious baseline?

If you filter the dataset to the top-return trajectories and imitate only those, you get a strong supervised baseline. DT must beat more than naive behavior cloning; it must show that return conditioning and sequence context add value.

7.Limitations and failure modes

Decision Transformer is elegant, but its assumptions matter.

  • Data coverage is everything. If the offline log never visits the state-action region needed for high return, the model has little basis for acting there.
  • Return conditioning may not stitch perfectly. The dream is to combine good sub-trajectories from many mediocre rollouts. In practice, the model may imitate whole trajectory styles rather than compose the globally optimal path.
  • No built-in online exploration. The basic DT setup has no mechanism like curiosity, entropy bonuses, or optimism to seek unknown states.
  • Target return is an extra knob. Standard RL does not require you to prompt with an expert return; DT usually benefits from knowing a reasonable desired return scale.
  • Context length still limits memory. Attention helps only over tokens the model can actually see.
The offline-RL danger

A model trained on logged data can be confidently wrong out-of-distribution. In safety-critical settings, DT should be paired with uncertainty estimation, conservative objectives, validation rollouts, or online fine-tuning safeguards.

✓ Check yourself — what is the biggest missing ingredient for online RL?

Exploration. A deployed learner must decide when to try unfamiliar actions to gather better data, but the basic Decision Transformer is trained as a supervised predictor over a fixed dataset.

9.Worked numeric micro-example: return-to-go

Take a short trajectory with rewards:

$$r=[1,0,2,1].$$

The return-to-go at time \(t\) is the reward still available from \(t\) onward:

$$\hat R_t=\sum_{t'\ge t} r_{t'}.$$

Compute it by running a sum from the right:

$$ \begin{aligned} \hat R_4 &= 1,\\ \hat R_3 &= 2+1=3,\\ \hat R_2 &= 0+2+1=3,\\ \hat R_1 &= 1+0+2+1=4. \end{aligned} $$

So the return-to-go sequence is:

$$\hat R=[4,3,3,1].$$

To make the model act well, you prompt it with a high target return-to-go — say 4 — as the first token. Prompt it with 2, and it tends to act like a mediocre agent, because you asked for behavior associated with a lower return.

✓ Check yourself — what is the return-to-go sequence for rewards [0,0,5]?

From the right: \(\hat R_3=5\), \(\hat R_2=0+5=5\), \(\hat R_1=0+0+5=5\). So \(\hat R=[5,5,5]\).

10.End-to-end trace through Decision Transformer

StepWhat happensShape/sequence
1Start from an offline trajectory window of \(K\) timesteps.\((\hat R_1,s_1,a_1,\hat R_2,s_2,a_2,\ldots,\hat R_K,s_K,a_K)\)
2Embed each return-to-go, state, and action with its token-type embedding, then add a timestep embedding.Each token becomes a vector in \(\mathbb{R}^{d}\)
3Stack the embedded tokens in causal order.Sequence length \(3K\), hidden size \(d\): \(3K \times d\)
4Run the sequence through a GPT-style causal transformer.Each position attends only to earlier positions in the \(3K\)-token context
5At each state token, predict the next action \(a_t\).\(h(s_t)\rightarrow \hat a_t\)
6Execute the predicted action, observe reward and next state, update the remaining target, and append the new tokens.\(\hat R_{t+1}=\hat R_t-r_t\), then continue autoregressively
How attention helps credit assignment

Self-attention can link a far-future return token to the earlier state and action that helped earn it. Instead of relying only on bootstrapped value estimates to move reward backward one step at a time, the transformer can learn long-range trajectory patterns directly inside its context window.

✓ Check yourself — can you summarize DT in one breath?

Decision Transformer turns offline RL trajectories into sequences of return-to-go, state, and action tokens, trains a causal transformer with supervised action prediction, and at test time prompts it with a desired return so it generates actions consistent with that level of performance.

Glossary

TermPlain meaning
AgentThe decision-maker that observes states and chooses actions.
EnvironmentThe world or simulator that receives actions and returns next states and rewards.
State \(s_t\)The information available at time \(t\), such as a game screen or robot sensor vector.
Action \(a_t\)The choice made by the agent at time \(t\).
Reward \(r_t\)The scalar feedback signal after an action.
Policy \(\pi\)The rule or model mapping observations to actions.
ReturnTotal future reward; the objective RL tries to maximize in expectation.
Return-to-go \(\hat R_t\)The sum of rewards still remaining from time \(t\) onward.
Offline RLLearning a policy from a fixed dataset without additional environment interaction during training.
Behavioral cloningSupervised imitation learning: predict the dataset action from observations.
TD-learningTemporal-difference learning; updates values using bootstrapped estimates of future value.
Bellman backupThe recursive value update that relates current value to reward plus future value.
CQLConservative Q-Learning, a strong offline-RL method designed to avoid overestimating unseen actions.
Trajectory TransformerA contemporaneous transformer approach that models trajectories for planning.
