Retrieval-Augmented Language Models

Douwe Kiela's lecture shows why language models should not be forced to memorize the world in their weights: retrieval turns a closed-book generator into an open-book system with updateable evidence, citations, and better grounding.

🎓 Douwe Kiela / Contextual AI ⏱ ~50 min ▶ Watch
Why this matters

A plain language model stores facts parametrically: inside billions of weights fixed by training. That is powerful, but awkward for enterprise use: facts go stale, private information is hard to remove, attribution is unclear, and the model can answer with high-confidence fiction. Retrieval augmentation adds an external memory the model can look up at answer time.

Running analogy

Think of the model as a student taking an exam. A closed-book student must cram every fact into memory; an open-book student can look up the right page, quote it, and then reason from it. In this note, the weights are the student's memory, the index is the book, the retriever finds pages, and the generator writes the final answer.

Prerequisites — 30-second refresher

You'll follow the lecture smoothly if these four pieces are fresh.

  • Language model generation — an LM predicts the next token from previous tokens; RAG changes the conditioning context before generation.
  • Embeddings and dot-product/cosine similarity — texts become vectors; aligned vectors have high dot product or cosine similarity.
  • Nearest-neighbor / vector search — given a query vector, find the closest document vectors quickly, often approximately.
  • Encoder-decoder model — an encoder reads input context and a decoder generates output; several RAG systems fuse retrieved evidence through this split.

1.Closed-book LMs break in the real world

Start with a customer support bot. Yesterday the refund policy changed. A closed-book LM may still answer from whatever was true at training time, and even worse, it may sound certain. Douwe frames this as a systems problem, not merely a model-size problem: the model needs a way to consult current, trusted evidence.

1.1 Parametric vs non-parametric memory

Parametric knowledge lives in neural weights. It is compressed, fast at inference, and useful for general language and reasoning patterns. But updating one fact can require retraining, fine-tuning, or fragile model editing. Non-parametric knowledge lives outside the model, in a corpus or index. You can add, delete, replace, and audit it without rewriting the whole student.

Key equation — closed-book generation

$$p_\theta(y\mid x)=\prod_{t=1}^{T}p_\theta(y_t\mid x,y_{<t})$$

Here \(x\) is the prompt, \(y\) is the generated answer, and \(\theta\) are the LM parameters. In a closed-book setting, all factual support must be recoverable from \(\theta\) and the prompt alone.

1.2 RAG as the open-book move

Retrieval-augmented generation changes the contract: before answering, retrieve a small set of relevant documents \(D_k\), then generate conditioned on both the question and those documents. That gives the system an external memory whose contents can be updated, versioned, and cited.

Common misconception

"Just make the model bigger to know more facts." Bigger parametric memory helps, but it does not solve staleness, deletion, attribution, or domain customization. The open-book move separates knowing where to look from writing the answer.

✓ Check yourself — why is deleting a fact easier in an index than in weights?

Because an index contains explicit records or chunks. You can remove or replace the offending document and rebuild or refresh the index. In weights, the fact is entangled with many other learned patterns, so targeted deletion is hard to verify.

2.Retriever + reader/generator

The smallest honest RAG system has two jobs. First, the retriever finds likely evidence. Second, the reader/generator uses that evidence to answer. In the lecture, Douwe emphasizes that today's common "frozen RAG" cobbles together independently trained pieces; the research frontier is making the pieces learn as a system.

2.1 The pipeline

For a question like "What changed in our refund policy?", the retriever should find the policy document, not a random FAQ. The generator then writes a concise answer grounded in those retrieved chunks.

Key equation — retrieval-augmented generation

$$p(y\mid x)\approx \sum_{d\in \mathrm{TopK}(x)} p_\eta(d\mid x)\,p_\theta(y\mid x,d)$$

\(p_\eta(d\mid x)\) is the retriever's probability for document \(d\), and \(p_\theta(y\mid x,d)\) is the generator's answer probability when that document is supplied. RAG marginalizes over retrieved documents instead of trusting only one memory trace.

2.2 Sparse, dense, and hybrid search

The lecture starts retrieval from TF-IDF and BM25: sparse methods count word overlap but downweight common words like "the". Dense retrieval uses learned vectors, so it can match semantic neighbors and synonyms. In practice, Douwe notes that many production systems use hybrid search: cast a wide lexical net, then refine or fuse with dense scores.

RetrieverWhat it storesStrengthFailure mode

✓ Check yourself — why might BM25 beat dense retrieval for a product code?

A product code is an exact lexical identifier. Dense models may map it near semantically related but wrong items, while sparse matching preserves the precise token overlap.

3.Dense retrieval by inner products

Imagine each document as a point in a high-dimensional library map. A query is another point. Dense retrieval asks: which document points lie closest to the query point?

3.1 DPR-style dual encoders

DPR-style systems use two encoders: one for the query and one for documents. They can be BERT-like encoders, but the important systems idea is the bi-encoder: encode all documents once, cache them in an index, then encode the query at runtime.

Key equation — dense retrieval score

$$\text{score}(q,d)=E_q(q)^\top E_d(d)$$

\(E_q(q)\) is the query embedding and \(E_d(d)\) is the document embedding. If vectors are normalized, this dot product is cosine similarity; otherwise it is a maximum-inner-product-search score.

3.2 ANN indexes and FAISS

A corpus may have millions or trillions of chunks, so exact comparison with every vector can be too slow. Approximate nearest-neighbor search trades a little recall for speed. The lecture highlights FAISS as the foundational open-source library behind many vector databases: very fast inner-product and nearest-neighbor search, often with compression or quantization tricks.

Why the bi-encoder is fast

The expensive document encodings are cached. At test time, the system computes one query vector and searches precomputed document vectors. That is why dense retrieval can be bolted onto a generator as an external memory.

✓ Check yourself — what do you lose when you move from a cross-encoder to a bi-encoder?

You lose rich token-level interaction between the query and each document during first-stage retrieval. In exchange, you gain the ability to precompute document vectors and search huge corpora efficiently.

4.Systems in the lecture: from open-book QA to RAG 2.0

Douwe's arc is historical and architectural: early open-domain QA separates retrieval and reading; dense retrievers improve semantic search; then systems like REALM, RAG, FiD, RETRO, and Atlas ask how retrieval and generation should be trained together.

4.1 REALM, RAG, and Fusion-in-Decoder

REALM is the visionary early system: retrieval happens during pretraining, and the model can update retrieval components, including document-side representations through expensive index refreshes. RAG combines a dense retriever with a seq2seq generator and trains by marginalizing over top-k documents. Fusion-in-Decoder (FiD) handles more passages by encoding each retrieved document separately, then letting the decoder fuse them.

Key equation — training objective with retrieved evidence

$$\mathcal{L}(\theta,\eta)=-\log\sum_{d\in \mathrm{TopK}(x)}p_\eta(d\mid x)p_\theta(y\mid x,d)$$

The loss rewards retrievers that put useful documents in the top-k and generators that use those documents to predict the target answer \(y\). The hard part is that top-k retrieval is discrete, so many systems approximate, marginalize, rerank, or refresh indexes.

4.2 RETRO, Atlas, and newer RAG

RETRO retrieves at the chunk level during language-model pretraining, using external neighbors so a smaller model can lean on a large datastore. Atlas studies few-shot retrieval-augmented language modeling and compares retriever-training strategies such as attention distillation, RePlug-style KL losses, and leave-one-out document utility. The lecture also names kNN-LM, RePlug, FLARE, Self-RAG, SILO, LENS, HyDE, LlamaIndex, LangChain, Chroma, Weaviate, and reranking systems as signs that RAG is becoming a full systems discipline.

Common misconception

"RAG" only means pasting search results into a prompt. That frozen version is useful, but Douwe calls it closer to Frankenstein's monster: pieces stitched together without joint optimization. The deeper research question is how to train retrieval, ranking, fusion, and generation to cooperate.

✓ Check yourself — why can FiD scale to more documents than naive concatenation?

It encodes each retrieved passage separately, so the encoder does not need to process one huge concatenated sequence. The decoder then attends across the encoded passage representations and fuses the evidence.

5.Training and trade-offs

Once retrieval is part of the model, training becomes a design space. Do you freeze the generator? Train only a reranker? Update the query encoder? Refresh the document encoder? Retrieve once, every token, every chunk, or only when uncertainty is high?

5.1 Frozen vs end-to-end

Frozen RAG uses an off-the-shelf embedder, vector database, and black-box LM. It is easy to deploy, but the retriever is not trained for the generator's needs. Retriever-tuned RAG keeps the LM fixed but backpropagates into a retriever or reranker using likelihood, KL, or REINFORCE-style signals. End-to-end RAG updates the generator and retrieval components together, but document-encoder updates can require costly index refreshes.

Key equation — turning scores into retrieval probabilities

$$p_\eta(d_i\mid q)=\frac{\exp(s_i)}{\sum_{j\in \mathrm{TopK}}\exp(s_j)},\qquad s_i=E_q(q)^\top E_d(d_i)$$

The softmax makes retrieved documents compete. A document with a higher inner product receives more probability mass, so it contributes more to the marginalized answer likelihood.

5.2 Benefits and limits

The upside is concrete: updateable knowledge, citations and attribution, smaller models supported by large indexes, domain customization, and less hallucination when the generator is trained to respect evidence. The limits are equally concrete: the retriever bounds the whole system, retrieval adds latency, documents can be stale or conflicting, context windows can still overflow, and fusing many pieces of evidence is hard.

Hallucination is not merely "being wrong"

Douwe distinguishes hallucination from generic mistakes. In a grounded setting, hallucination means the answer conflicts with the evidence or ground truth the system was supposed to use. RAG helps because it gives a concrete source of truth, but a generator can still ignore or misread retrieved text unless the system is trained and evaluated for grounding.

✓ Check yourself — name two reasons long context does not make retrieval obsolete.

First, attending over an entire book to answer one fact is inefficient. Second, long-context models still need to decide which parts matter; scalable attention often becomes sparse or retrieval-like behind the scenes.

6.Worked micro-example — dense retrieval scoring

Now compute the smallest honest dense-retrieval step by hand. Suppose the query embedding \(q\) is compared with three document embeddings \(d_1,d_2,d_3\). The inner products are already computed:

Step-by-step with real numbers

$$ q\cdot d_1=8.0,\qquad q\cdot d_2=2.0,\qquad q\cdot d_3=5.0 $$

Convert scores into a retrieval distribution with softmax. For numerical clarity, subtract the largest score, 8.0:

$$ \operatorname{softmax}([8,2,5])=\frac{[e^{0},e^{-6},e^{-3}]}{e^{0}+e^{-6}+e^{-3}} =\frac{[1,0.00248,0.04979]}{1.05227} $$

So the probabilities are approximately

$$ [p(d_1\mid q),p(d_2\mid q),p(d_3\mid q)]\approx[0.9503,0.0024,0.0473]. $$

Top-1 retrieval selects document 1. Top-k retrieval might keep documents 1 and 3, then let the generator fuse or marginalize their evidence.

✓ Check yourself — how does the answer change if you retrieve top-2 and fuse?

The top-2 set is \(\{d_1,d_3\}\). If you renormalize only over those two, \(p(d_1)=1/(1+e^{-3})\approx0.9526\) and \(p(d_3)\approx0.0474\). The generator can now use document 3 as backup, contradiction, or extra detail instead of relying entirely on document 1.

7.End-to-end trace — one question through RAG

Follow one concrete question through the full system. The key state changes from text, to vector, to document IDs, to a grounded prompt, to an answer with attribution.

StepWhat happensShape/Output
0 · questionUser asks: "What is our current refund window for annual plans?"text string
1 · query encoderEncode the question into a dense vector.embedding \(q\in\mathbb{R}^{d}\)
2 · ANN search over N-doc indexSearch cached document embeddings for maximum inner product or cosine similarity.scores over N docs
3 · top-k docsReturn the highest-scoring chunks, e.g., policy page, update memo, FAQ.\(D_k=\{d_1,\ldots,d_k\}\)
4 · concatenateBuild a prompt from the question plus retrieved documents and instructions to cite evidence.query + docs context
5 · generator/readerGenerate an answer while attending to the retrieved evidence.tokens \(y_1\ldots y_T\)
6 · grounded answerReturn the answer and link it to the supporting document.answer + citation
Parametric vs non-parametric knowledge

The parametric part is the generator's language ability and general reasoning in weights. The non-parametric part is the index: explicit, swappable evidence. Retrieval reduces hallucination because the answer is no longer forced to emerge from weights alone; it can be constrained by a document the system can cite, inspect, update, or delete.

✓ Check yourself — where can this trace fail even if the generator is excellent?

At retrieval. If the ANN search returns the wrong chunks, the generator is grounded on bad evidence. RAG quality is bounded by retrieval quality, document freshness, ranking, and how well the generator fuses the selected context.

Glossary

TermPlain meaning
Parametric knowledgeFacts or patterns stored implicitly in model weights.
Non-parametric memoryExternal documents or indexes the model can retrieve from at runtime.
RAGRetrieval-augmented generation: retrieve evidence, then generate conditioned on it.
RetrieverThe component that maps a query to relevant documents or chunks.
Reader/generatorThe model that consumes the question plus retrieved context and produces the answer.
BM25A strong sparse lexical retrieval method based on weighted term overlap.
DPRDense Passage Retriever; a dual-encoder approach for semantic passage search.
MIPSMaximum Inner Product Search: find vectors with the largest dot product against a query.
ANNApproximate nearest-neighbor search; fast approximate vector lookup over large indexes.
FAISSFacebook AI Similarity Search, a widely used library for efficient vector search.
FiDFusion-in-Decoder; encode documents separately and fuse evidence in the decoder.
Index refreshRe-encoding and rebuilding document vectors after the document encoder or corpus changes.
GroundingConstraining an answer to match explicit evidence or a trusted source.
